# herdr 方式B: main ペインの最小サイズ確保（分割起点の切替）

- 日付: 2026-06-27
- 起票: loop-doctor（ユーザー指摘）
- 対象 run: なし（構造的不具合。指示文 `loop-runner/SKILL.md:34` の分割ロジックそのものが原因）
- 承認: ユーザー（選択肢 A = 推薦1のみ）

## 症状
方式B（並列タスクを herdr ペインで可視化起動）でタスクを起動するたびに、オーケストレータ（main）
ペインが 1/2 → 1/4 → 1/8 と縮む。最大3タスクで main が最小 1/8 まで小さくなり作業しづらい。
要望: main は最低でも画面の 1/4 を維持。タスクペインはスペースが無ければ縮んでよい。

## 根本原因
分割の起点が常に main ペインだったこと（`herdr pane split <自分のpane> ...`）。herdr の split は
対象ペインを既定 0.5 で二分割するため、main を繰り返し分割すると main が等比で縮む。

## 変更
- 対象ファイル: `.claude/skills/loop-runner/SKILL.md`（方式B、旧 line 33-37）。
- 内容: 分割の起点を切り替え。
  - 1本目のタスク: main を `--direction right` で分割し「タスク列」を作る。生成ペインを `ANCHOR` に記録。
  - 2本目以降: main ではなく直前タスクペイン `$ANCHOR` を `--direction down` で分割して右列に積む。
- 効果: main は台数に依存せず約 1/2 を維持（要件の 1/4 を割らない）。タスクペインだけが右列で縦に縮む。
- stage-runner は独自の split を持たず方式Bに委譲しているため、この1箇所の修正で stage-runner も是正される。

## 副作用（許容済み）
タスクペインが右列に縦積みになり、台数が増えると1枚ずつ小さくなる（ユーザー許容）。
`herdr pane read` はサイズ非依存で影響なし。

## 検証
次回の方式B 並列 run（loop-runner / stage-runner）で、タスク2本・3本起動後も main ペインが
1/4 以上を保つことを目視確認する。
