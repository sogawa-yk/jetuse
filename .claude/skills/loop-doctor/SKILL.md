# loop-doctor：ループ自己改善

あなたが直すのは**コードではなく「ループの仕組み」**である。
対象は SKILL.md / loop-config.yml / goal 完了条件 / hooks / STATE.md 設計 / settings.json。

## 手順
1. 人間の指摘を「観測された症状」として箇条書きに正規化する。
2. `references/component-map.md` を読み、症状→疑う構成要素→典型的な編集対象を引く。
3. **履歴で裏を取る（推測で診断しない）。** 直近 run の
   `runs/<id>/reviews/`・`turns/`・`manifest.json` を読み、症状を生んだ証跡を特定する。
   - 「レビューが甘い」→ 該当 `review-*.json` にその問題の指摘が**無い**ことを確認。
   - 「終わらない／空回り」→ `turns/*.json` の `goal_checker.reason` の繰り返しを確認。
   - 「トークン浪費」→ `manifest.totals` と各 turn の差分サイズ・レビュー回数を確認。
   - 履歴が大量なら `log-summarizer` サブエージェントに要約させてから読む。
4. 根本原因を1〜3個に絞り、各々に「どのファイルを・どう変えるか」を付けた推薦を、
   効果と副作用とともに優先順位付きで提示する。
5. **ここで停止し、人間の承認を待つ（ヒューマンゲート）。勝手に編集しない。**
6. 承認された項目だけ編集する。変更内容・理由・対象 run・対象ファイルを
   `runs/_meta/improvements/<date>-<topic>.md` に追記する。
7. 「次の run で検証してください」と伝える。改善が効いたかは次ループの履歴で判断する。

## 提示フォーマット（ALWAYS この順）
### 症状
### 履歴上の証跡（ファイルパス付き）
### 根本原因の仮説
### 推薦（優先順位・対象ファイル・具体的変更・想定副作用）
### 承認待ち（どれを適用しますか？）

> 注意: loop-doctor は「思考を避ける口実」にもなり得る。推薦の**根拠ログを必ず読む**こと。
> 証跡なしの推薦は出さない。
