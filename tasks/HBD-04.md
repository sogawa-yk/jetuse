# タスク: HBD-04 合成バリデーション（ガバナンス：許可組合せ・必要ケイパ・権限スコープ）

## ゴール
合成したデモ構成を、デプロイ前に**ガバナンス4制約（§4）**で検証する。許可された組合せ・必要ケイパビリティ・
権限スコープに反する構成を弾き、外れたデモを作らせない。

## 対象 area
api（バリデーション中核）。web は検証結果の提示。

## 依存
HBD-03（デモ構成オブジェクト）。manifest の `permissions`/`requires`（PLG-01）と合成バリデーションの土台（SBA-01）を再利用。

## 受け入れ条件（検証可能な述語で書く）
- [ ] バリデータ: デモ構成に対し（a)許可組合せ（sample-app×AI部品×connector の対応表）(b)必要 capability が束縛済み (c)権限スコープが manifest `permissions` 内 (d)モデル可用性（ap-osaka-1）を判定する
- [ ] 違反は機械可読な結果（違反種別・該当要素・代替提案）で返し、**代替提案（外させない）**を含む
- [ ] 既存の合成バリデーション土台（SBA-01 / `validate_composition` 相当）と整合し、二重定義しない
- [ ] 正常構成は PASS、各違反種別（許可外組合せ／未束縛ケイパ／スコープ逸脱／モデル不可）が個別にFAILする単体テストを網羅
- [ ] api lint（ruff）・既存テスト後方互換クリーン

## E2E シナリオ（実環境 / jetuse-dev・複数）
完了ゲートで loop 環境へデプロイ（DB は専用スキーマ `JETUSE_HBD-04` で隔離）し、証跡を `runs/<run-id>/e2e/` に残す。
- [ ] シナリオ1（正常）: 妥当な構成（例 SBA-A＋RAG-QA＋Slack）が PASS し、デプロイ前ゲートを通過する
- [ ] シナリオ2（違反→代替）: 許可外組合せ／権限スコープ逸脱の構成が FAIL し、代替提案が返る（実 GenAI 補助で文章化する範囲含む）
- [ ] シナリオ3（能力前提）: VLM 等の不可用ケイパを要求する構成が「能力不足」で弾かれ警告される
- [ ] 実施不能な範囲は `runs/<run-id>/e2e/SKIPPED.md` に理由明記
