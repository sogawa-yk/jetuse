## 作業内容
- `specs/18-sp2-demo-tenancy.md` を起草する。最低限決めること:
  - **Demo エンティティ完全形**: 列（config の形式・status の状態遷移 `provisioning → ready / failed` 等・
    updated_at）、usecases の owner_sub / visibility パターンとの整合、016_demos.sql（SP1-02）からの差分。
  - **CRUD API 契約**: パス（`/api/demos` 配下）・認可（require_user / require_demo / require_demo_owner の
    使い分け）・存在秘匿 404 方針（SP1 踏襲）・レスポンス形。
  - **箱のプロビジョニング方式**: `demo_<id>` DB スキーマ生成（BE-02 のサンプルスキーマ実体化機構の流用範囲）、
    RAG ベクタストア（既存 ensure_store の lazy 生成をデモ箱にどう適用するか）、
    削除時の後始末（スキーマ drop・ストア削除・登録簿行・会話紐付けの順序と fail-safe）。
  - **テナンシ分離（Identity Domains）**: アプリ側要件（OIDC 設定契約・AUTH_REQUIRED 既定）と
    **人間側作業の一覧**（Identity Domain アプリ登録・テストユーザー・メタデータ提供 — IAM/IdP は人間ゲート）。
  - **SP2-01〜04 の受け入れ条件の確定**（tasks/SP2-0X.md と STAGE2-PROGRESS.md の更新も本タスクの成果物）。
- 判断が割れる論点（例: プロビジョニングを同期/非同期どちらでやるか）は選択肢と推奨を明記し、
  必要なら `docs/comparison/` に比較ドキュメントを添える（比較ドキュメント主義）。
