# タスク: SP3-07 GitHub Actions による jetuse:dev 自動デプロイ

## 目的
**dev から派生した開発用ブランチへの push で、GitHub Actions が jetuse:dev コンパートメントの
プレビュー環境（RM スタック `jetuse-dev-app`）へ自動デプロイする**パイプラインを作る
（施主指示 2026-07-09。以降の実環境 E2E はこのデプロイ環境で行う方針への転換）。

## 背景（確定事実）
- プレビュー環境 = RM スタック `jetuse-dev-app`（API Gateway + Container Instance + SPA バケット、
  gateway https://jstvwfl2fhsx55p2zaubm5ui6m.apigateway.ap-osaka-1.oci.customer-oci.com、AUTH オフ、
  ADB=loop ADB 再利用、GenAI はリソースプリンシパル）。
- **RM の apply/plan は `oci resource-manager job` 経由なら通る**（SP2 実績。ローカル terraform apply は不可）。
- `infra/terraform/environments/dev-app/` の .tf は**リポジトリにあるが環境実値（tfvars）は未コミット**。
  実値は RM スタック側の変数に保持。
- これまでのデプロイはループが手動（イメージ build → OCIR push → RM 変数更新 → apply job）で実施。
  OCIR push は jetuse-dev グループ権限（memory: ocir-compartment-layout）。
- 既存 CI: .github/workflows の api / terraform / web チェック（PR 時）。

## 対象 area
infra（.github/workflows + infra/terraform/environments/dev-app）+ docs

## 作業内容
1. **デプロイワークフロー** `.github/workflows/deploy-dev.yml`:
   - trigger: `push` to **dev 派生の開発用ブランチ**（`main` / `dev` / `internal-stable` は除外。
     `feat/**`・`fix/**`・`chore/**` 等のパターンで表現）。
   - **concurrency 制御必須**: 共有スタック 1 本のため group=dev-app-deploy・cancel-in-progress
     （並行 push の apply 競合を構造で防ぐ）。
   - steps: checkout → API イメージ build → OCIR へ push（tag = `<branch>-<short-sha>`）→
     web SPA build → SPA バケットへ同期 → RM スタック変数（image_url 等）更新 →
     RM plan+apply job 起動 → 完了待ち（失敗時はログを step summary へ）→
     **smoke テスト**（gateway の /api/health 200・SPA 200・モデル一覧に gpt-5 系が出る）→
     step summary に gateway URL とデプロイ内容を出力。
   - OCI 認証は GitHub Secrets（OCI CLI 標準 env: OCI_CLI_USER/TENANCY/FINGERPRINT/KEY_CONTENT/REGION
     + OCIR 用 auth token）。**必要な Secrets の一覧と値の入手方法を README 化**し、最終メッセージにも
     列挙する（Secrets 登録は人間ゲート — 値は絶対にリポジトリへ入れない）。
2. **dev-app スタックの IaC 整備**: CI から更新する変数（image_url・SPA 内容）と、SP3-06 で増えた
   実行時 env（GEN_SHARED_PROFILE / GEN_SHARED_COMPARTMENT_OCID、GENERATION_PROXY_URL、
   ORASEJAPAN 認証材料の受け渡し方式 — RM の sensitive 変数 → コンテナ env を想定）を
   dev-app tf で宣言的に扱えるよう更新。**秘匿値の実値はコミットしない**（RM 変数に置く。
   設定手順を docs へ）。sign_proxy を配備像でどう動かすか（API プロセス内 mount or 同一 CI 内
   別コンテナ）も本タスクで確定・実装する。
3. **ローカルからの初回デプロイ検証**: GitHub Secrets 登録前でも検証できるよう、ワークフローと
   同一手順のスクリプト（ops/deploy-dev-app.sh 等・冪等）を用意し、**実機で 1 回デプロイして
   feat/SP3-06 相当のコードが gateway で動くことを確認**（= E2E）。
4. docs/guides/dev-environments.md へ「開発ブランチ push → 自動デプロイ」の運用を追記。

- [ ] review_verdict=PASS。

## E2E シナリオ
- ops/deploy-dev-app.sh（ワークフロー同一手順）で jetuse:dev へ実デプロイ → gateway で
  SPA・/api/health・ビルダー UI モデル一覧を確認。既存デモの閲覧・chat 実応答 1 本。

## 非ゴール / 禁止事項
- 生成ランタイムの実 CI 化（SP3-08）。main/dev への push トリガー・本番リリースフロー。
- GitHub Secrets の実値登録（人間）。IAM 変更。認証情報・OCID 実値のコミット禁止。
- 共有 RM スタックの破壊的変更（destroy 禁止。apply は本タスクの検証に必要な範囲のみ）。
