# タスク: SP3-01 ビルダー・パイプライン API 骨格 + ヒアリング(NL)

> 受け入れ条件は **specs/19 §2・§8（+ §9 SP3-01）で確定**(SP3-00 起草済み。
> **specs/19 の人間承認が着手条件**)。

## 目的
ビルダーの背骨となる API を敷く。フィールドSA が自然言語でヒアリングに答え、その内容を
「デモ設計に渡せる構造化された要求サマリ」へまとめるところまで。以降の段（設計/生成/データ）が乗る土台。

api

## 作業内容（specs/19 で確定）
- `builder_sessions` エンティティ（specs/19 §2.1 — 専用エンティティ・単文 migration 2 ファイル・
  状態は `hearing`→`designed` の 2 つのみ・owner_sub は識別列で WHERE 句強制）。
- ヒアリング API 3 ルート（specs/19 §2.4）: `POST /api/builder/sessions` /
  `POST .../{sid}/messages`（LLM 構造化出力・同期 JSON）/ `GET .../{sid}`。
- 要求サマリ（specs/19 §2.2）と「設計に足りる」判定（specs/19 §2.3 —
  **LLM 判定の上にサーバ側の決定的再検査**。fail-closed）。
- 信頼境界の入力上限（specs/19 §2.1 — 発話 4,000 文字・50 往復 / 256KB で 422）。
- **ready ゲートの導入**（specs/19 §8.1 — デモスコープ能力ルート・`/app/` 配信の共通依存を
  「`status='ready'` 以外は 404」へ一般化。SP2 までの挙動は不変 = 回帰テストで担保）。
- 単体テスト: fake LLM でのヒアリング往復（不足→追質問 / 充足→sufficient）・決定的再検査・
  越境 404（存在秘匿）・401・上限 422・demo_id 設定後の読み取り専用 409・ready ゲート回帰。

## 完了条件（検証可能な述語で — specs/19 §9 SP3-01 と同一）
- [ ] builder_sessions migration が fresh 適用・冪等再適用とも成功（ランナー事後条件検証の対象に追加）。
- [ ] §2.4 の 3 ルートが動作し、NL 応答から**構造化要求サマリ**を産出。サーバ側決定的再検査が効く
  （LLM が sufficient=true でも必須欠落なら false）。
- [ ] 所有者以外はセッション不可視（404 存在秘匿）。単体テスト全緑・SP2 回帰なし・ruff クリーン。
- [ ] 実環境 E2E（プレビュー環境 + 実 LLM）: 1 セッションのヒアリング往復 → sufficient=true 到達を
  `runs/<run-id>/e2e/` に記録。

## E2E シナリオ
- プレビュー環境（STAGE3-PROGRESS「E2E 方針」の gateway）に対し、セッション作成 → NL 応答投入
  （不足 → 追質問）→ 充足 → sufficient=true・要求サマリ産出までを実機で確認。
