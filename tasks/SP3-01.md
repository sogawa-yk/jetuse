# タスク: SP3-01 ビルダー・パイプライン API 骨格 + ヒアリング(NL)

> 受け入れ条件は specs/19 準拠で確定（SP3-00 起草。**specs/19 の人間承認が着手条件**）。以下は暫定。

## 目的
ビルダーの背骨となる API を敷く。フィールドSA が自然言語でヒアリングに答え、その内容を
「デモ設計に渡せる構造化された要求」へまとめるところまで。以降の段（設計/生成/データ）が乗る土台。

## 仕様参照
specs/19-sp3-builder.md（パイプライン・ヒアリング契約）、specs/17 §5・§6、SP1 の DemoContext seam。

## 対象 area
api

## 前提（依存タスク / 人間の事前作業）
SP3-00（specs/19 承認）。

## 作業内容（specs/19 で確定・暫定）
- ビルダーセッションの状態を持つ（会話 or 専用エンティティ。所有者 = 認証ユーザー・fail-closed）。
- ヒアリング API: セッション開始 / 応答投入（NL）/ 現在の収集状態取得。LLM で要求を段階的に構造化し、
  「設計に十分か」を判定する。収集結果 = **要求サマリ**（業種・ユースケース・使う能力候補・データの雰囲気）。
- 認可は require_user ベース。ビルダーは Internal 専用（AUTH_REQUIRED=true 前提の面）。
- 単体テスト: セッション往復・越境404（他人のセッション不可視）・要求サマリの構造検証・
  「情報不足→追質問 / 十分→設計可能フラグ」の分岐。

## 完了条件（検証可能な述語で）
- [ ] ヒアリング API が動作し、NL 応答から**構造化要求サマリ**を産出する。
- [ ] 所有者以外はセッション不可視（404 存在秘匿・SP1/SP2 の流儀）。単体テスト全緑・回帰なし・ruff クリーン。
- [ ] 実環境 E2E（デプロイ済み環境）: 1 セッションのヒアリング往復→要求サマリ産出を `runs/<run-id>/e2e/` に記録。

## E2E シナリオ
- デプロイ済み環境（gateway）に対し、ヒアリング1往復以上→設計可能判定までを実機で確認。
