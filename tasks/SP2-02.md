# タスク: SP2-02 箱のライフサイクル（lazy 解決・削除後始末・VPD 基盤）

> 受け入れ条件は specs/18 §3 準拠で確定（SP2-00 起草。**specs/18 の人間承認が着手条件**）。
> specs/18 §3.1 の決定（論理名前空間 + lazy 生成）により、本タスクの本体は**削除の後始末**。
> デモ作成時の eager な箱生成は**実装しない**（POST は SP2-01 の INSERT のみで完成形）。

## 目的
デモの箱（RAG store / datasets 表 / 会話）のライフサイクルを閉じる:
lazy 生成（既存機構の流用で成立していること）の実機確認と、
`DELETE /api/demos/{id}` の後始末（順序付き・冪等・fail-safe）の実装。残骸を残さない。

## 仕様参照
specs/18-sp2-demo-tenancy.md §3（箱の方式・削除手順・同期採用）、
docs/comparison/demo-box-provisioning.md、specs/17 §5・§6

api

## 作業内容
- **`DELETE /api/demos/{id}` を後始末込みで初公開**（specs/18 §2.1 — SP2-01 では公開していない。
  同期 — §3.3 採用案）。specs/18 §3.2 の手順:
  1. 排他リース取得 → `deleting` へ遷移し commit。リースは **commit を跨いで保持できる方式**
     （DBMS_LOCK — FOR UPDATE は遷移 commit で失効。**不可なら 503 + ADR 差し戻しで
     フォールバックは無い**。specs/18 §3.2.1）で**手順 5 の完了まで保持**（DELETE 同士も直列化）
  2. DB 箱: **JETUSE_DATASETS の exact owner 一致行**を列挙して DROP TABLE PURGE
     （ORA-00942 成功扱い）→ 行削除 → プロファイル DROP。前提改修（specs/18 §3.2 手順 2）:
     **dataset 作成の registry-first 化**（登録 commit〔state='creating'〕→ CREATE TABLE →
     VPD → GRANT → state='ready'。一覧/プロファイルは ready のみ参照・creating 幽霊行は
     次回操作/DELETE 時に回収 — 未登録表を構造的に排除し、短縮 tag の接頭辞走査を削除根拠に
     しない。state 列は `_ensure_meta` の冪等 ALTER + 既存行 ready backfill — specs/18 §3.2）、
     **demo の datasets 命名（プロファイル・表接頭辞）の完全ハッシュ化**（8hex 衝突は
     object_list 経由のメタデータ漏えい + 削除波及 — specs/18 §3.2 手順 2）、
     **delete_dataset の DROP 先行化**（現実装は行先行削除+DROP 例外全無視 →
     「DROP 先行・ORA-00942 のみ成功扱い・他失敗は行を残して 503」へ）
  3. RAG 箱（**全バックエンド・exact な列挙**。対象名は各バックエンドの既存命名関数に
     `delete_owner(namespace)` 相当の公開関数を設けて取得 — 命名の再実装禁止・specs/18 §3.2）:
     rag_files 行列挙 → DP files/vs.files 削除（NotFound 成功扱い）→ 行削除 →
     **CP 一覧から `metadata.owner == sha1(owner キー) 40hex`（固定長完全ハッシュ — 作成側と
     同じ導出。specs/18 §3.2 手順 3b）でも列挙**して `vector_stores.delete`
     （NotFound=成功。**失敗は中断・503** — 枠漏れ防止。未登録 store も回収）→
     **ledger の owner_key=namespace 全行（pending/confirmed）を解放**（reservation_id で
     File/原本を削除 or 不存在確認 → 行解放。事後条件 = 当該 owner の ledger 行ゼロ）+
     **DP files.list の `<sha1(namespace)>/` 接頭辞一致を削除**（ledger にも無い孤児の保険 —
     外部名は specs/18 §3.1 の `file_key(owner_key, reservation_id, ext)` 導出関数を
     作成・削除・fixture で共用。UI 表示は resolve_citation_filenames が DB の元名で
     解決するため影響なし）→
     rag_select_ai の profile/vector index DROP（不存在無視。**demo の命名は完全 sha1 へ改修** —
     sha1[:8] の衝突は index 共有=越境読取・削除波及になる。specs/18 §3.2 手順 3d）→
     rag_opensearch の namespace index 削除（**台帳に opensearch 行があれば設定の有効/無効に
     かかわらず保存 locator で削除・404 成功扱い**。作成後に endpoint を無効化した DELETE の
     E2E 付き — specs/18 §3.2）→
     Object Storage 原本 prefix `rag/<sha1(namespace)>/` を**ページネーション付き全列挙で削除
     （NotFound 以外の失敗は中断・503 — best-effort にしない。バケット versioning 無効を
     配備 preflight で必須確認）** →
     **`rag_stores` 行と `demo_backend_targets` 台帳行の削除は RAG 箱の掃除が全て成功した後**
     （途中 503/クラッシュ後の再 DELETE が旧 target の locator を参照できるよう保持。
     specs/18 §3.2 手順 3b）。**スキップ判定の正は台帳**（現在の enabled/バケット設定でなく、
     台帳に kind 行があれば設定不一致は 503。specs/18 §3.2）。
     **CP store 一覧・DP files 一覧もページネーション完走**（先頭ページのみ不可）
  4. 会話: **messages を明示チャンク（例 1000 行 + commit）で先に削除**し、ゼロ確認後に
     conversations の demo_id 行を同様にチャンク削除（CASCADE 任せは大量 message 1 会話で
     タイムアウト→全量ロールバック。チャンク commit で再 DELETE が続きから収束 —
     specs/18 §3.2 手順 4。外部リソース無し = demo 会話は OCI Conversation を作らない・§4.2）。
     **usage_log は保持**（監査/利用量の明示契約）
  5. demos 行削除 → commit（リース解放）→ `{"deleted": true}`
  途中失敗は段階を detail に含む 503。再 DELETE で収束（各ステップ冪等・exact な列挙が
  登録の有無に依存しない — 補償削除・cleanup record は持たない。specs/18 §3.2）。
- **lazy 生成との競合 = demo 単位の排他リース（specs/18 §3.2.1）**: demo 名前空間に外部リソース・
  DDL・登録行を作る**全操作**（upload 全段 / ensure_store / dataset 投入 / ensure_profile /
  demo 会話 INSERT）は、**専用 DB セッションの DBMS_LOCK（排他・commit を跨いで保持）**を
  操作の開始から完了まで保持する。**ADB での DBMS_LOCK 可否をタスク冒頭で実機確認**
  （**不可なら demo mutation/DELETE 経路を 503 で停止し ADR 差し戻し** — FOR UPDATE は
  DDL 暗黙 commit で失効するため代替リースにならない。specs/18 §3.2.1）。
  **実装契約**: lock ID は有効範囲内の決定値（ORA_HASH 範囲指定）・REQUEST の全戻り値処理・
  finally で RELEASE・**リース専用の小プール**（作業プールと分離 — 枯渇防止。上限到達 503）。
  **単位を明示**: DBMS_LOCK timeout は秒（目安 300）・oracledb call_timeout はミリ秒
  （目安 310_000。リース専用セッションのみ・返却時に既定復元）。取得後の status 再確認は
  **2 契約**（mutation = 行なし/deleting 404 ／ DELETE = 内部専用 allow_deleting で deleting を
  受理し後始末を再開 — 各段階の 503 後に同じ DELETE が完走するテスト付き。specs/18 §3.2.1）。
  **リース API は scope（demo_id）を明示引数で受け**、
  owner 文字列の接頭辞から推測しない。キー空間分離は **owner キー導出ヘルパーの単射
  エスケープ**（sub が `demo_`/`sub_` で始まる場合のみ `sub_` 前置。認証拒否はしない —
  specs/18 §3.2.1）。**箱あたり上限**（specs/18 §3.1 — files/datasets の設定上限・超過 422・
  受け入れ閾値 = 配備経路の実効タイムアウト実測の 1/2）を実装。
  チャットの SSE 本体はリースを跨がない。user 単位ルートは変更しない。
- **VPD 基盤（specs/18 §4.3 層1 — 本タスクが所有。codex review-7 B001: registry-first 作成が
  「CREATE TABLE → VPD → GRANT」の順で依存するため）**: タスク冒頭で ADB の DBMS_RLS 可否を
  実機確認（不可なら ADR 差し戻し）→ 人間承認のうえ初回セットアップ（コンテキスト・setter
  パッケージ・ポリシー関数・権限）→ 起動時完全性検証（不完全なら dbchat/datasets 経路 503）→
  コンテキスト set/clear 契約（接続取得ごと set・finally clear）。既存表への遡及は
  「コンテキスト設定配備 → 一括付与」の二段階。既存 user dataset ルートが継続動作する配備順。
- **demo の OpenSearch index 名も完全ハッシュ化**（16hex 衝突の検索混入・削除波及の排除 —
  specs/18 §3.2 手順 3e。既存 user 資産は現行名維持 = residual）。
- **アプリ全体の DP Files 総数上限 = 予約 ledger**（specs/18 §3.1 の完全 DDL —
  quota 行 FOR UPDATE で直列化・pending/confirmed・**外部 filename/object 名に
  owner_key + reservation_id を埋めて exact 照合**・起動時 reconcile は未登録 File/原本を
  削除して解放。**Files を作る全経路に適用**〔user `/api/rag/files` 含む。Public 既定は
  無制限=挙動不変・Internal で有効化〕）。**demo DELETE 手順 3a で対応 ledger 行も解放**
  （枠の即時回復 — E2E: 削除直後の再 upload 成功・反復後に ledger 件数 = files.list 実件数）。
  **Select AI 有効構成では原本 put は必須リトライ可能ステップ**（失敗で登録成立させない —
  put 失敗 → 再試行で検索/citation 回復の障害注入テスト。specs/18 §3.1）。
- **配備前移行 + 構成ドリフト耐性**（specs/18 §3.2・§3.2.1 の順序 = preflight 分類 → 人間承認 →
  クリーンアップ → 再検証+マーカー。**user 資産と分類された行は変更せず 503 維持のまま
  個別 ADR へ差し戻し** — rekey はしない）: 旧命名の既存 demo 資源のクリーンアップ移行
  （旧形式 fixture + user 資産併置の実 E2E 付き）と、**write-ahead 台帳 `demo_backend_targets`**
  （単文 migration 022 = CREATE TABLE + 023 = 索引。完全 DDL は specs/18 §3.2 —
  **完全 locator** = 種別 + region / compartment OCID / GenAI project OCID /
  OS namespace+bucket / OpenSearch endpoint〔秘密値除く〕を外部書き込み前に INSERT・追記型・
  複数 target 保持。記録あり・設定不一致の DELETE は 503。ランナー検証/fault-injection 対象は
  017〜024（024 = rag_files.filename の CHAR 化 — specs/18 §3.1）。
  region/project/compartment 変更後の削除テスト付き）。
- **個別ファイル削除の外部先行化**（specs/18 §3.2 — 既存 delete_file の「行先行・外部失敗無視」を
  「外部削除成功 → 行 DELETE → カウンタ解放」へ。**Select AI `$VECTAB` の反映は同期一択**
  （再同期して不存在確認 → 行削除。失敗/不可は 503 で保持 — specs/18 §3.2）。
  単体テスト: OCI File/detach・原本・OpenSearch・Select AI の各失敗で行と
  カウンタが保持され再試行で収束。E2E: 個別 DELETE 後に全外部資源の不存在 + 検索非ヒット）。
- **リースの再入契約**（specs/18 §3.2.1 — 最外層のみ取得・内側へトークン伝播。
  upload→ensure_store / dataset→profile 再構築のネスト試験）。
- RAG の lazy 生成（`ensure_store(ctx.namespace)`）と datasets の lazy 生成が namespace キーで
  そのまま箱として成立することの確認（コード変更が要る場合は specs/18 との差分を報告）。
- 単体テスト（DB/GenAI 層は fake）: 状態遷移・後始末の順序・途中失敗→再実行収束・
  store 削除失敗/原本削除失敗の中断(503)・NotFound の成功扱い・deleting 中の require_demo 404・
  **usage_log の保持**・上限超過 422・**並行系**（mutation 進行中の DELETE 待機・deleting 遷移後の
  mutation 404・**同時 2 本の DELETE**・クラッシュ相当の未登録残骸〔store・file・原本〕の回収・
  **creating クラッシュ 3 点〔registry 直後/CREATE 直後/ADD_POLICY 失敗〕の一覧非表示と回収**・
  **tag 強制衝突の 2 owner で NL2SQL メタデータ非混入・削除非波及**・完全ハッシュ名の分離・
  リース契約（REQUEST 異常戻り値・timeout・専用プール上限の同時 mutation）—
  specs/18 §3.2.1 のテスト要件）。
