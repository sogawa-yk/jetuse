# タスク: SP2-02 箱のプロビジョニング（demo_<id> スキーマ生成・削除後始末）

> **暫定チケット**: 受け入れ条件の確定は SP2-00（specs/18 承認）後。以下は specs/17 §5-6 からの骨子。

## 目的
デモ作成時に「箱」を実体化する: `demo_<id>` DB スキーマの生成（サンプルスキーマ実体化機構 = BE-02 の流用）。
削除時は箱を後始末して残骸を残さない。「箱の分け方」（specs/17 §5）の DB 部分を実装する。

## 仕様参照
specs/18-sp2-demo-tenancy.md（SP2-00 成果物）、specs/17 §5「箱の分け方」・§6

## 対象 area
api

## 前提（依存タスク / 人間の事前作業）
SP2-01（Demo CRUD と status 列）。

## 作業内容（骨子 — specs/18 で確定）
- デモ作成フローに箱生成を接続: `demo_<id>` スキーマ作成 → status `provisioning → ready / failed`
  （同期/非同期は specs/18 の決定に従う）。
- RAG の箱は既存 lazy 生成（`ensure_store(ctx.namespace)`）で足りることを確認し、specs/18 の決定を反映。
- `DELETE /api/demos/{demo_id}` の後始末: スキーマ drop・ベクタストア削除・rag_stores/rag_files 行・
  会話紐付けの削除（順序と fail-safe は specs/18。途中失敗でも再実行で収束すること）。
- 単体テスト: 状態遷移・後始末の冪等性（DB/GenAI 層は fake）。
