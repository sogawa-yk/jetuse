# タスク: SP2-01 Demo エンティティ本格化 + CRUD ルート

> 受け入れ条件は specs/18 §1・§2 準拠で確定（SP2-00 起草。**specs/18 の人間承認が着手条件**）。

## 目的
SP1-02 の最小 demos テーブル（所有権検証のみ）を一級エンティティに拡張し、Demo の CRUD を API として提供する。
SP3（ビルダー）が「生成したデモを保存する」着地先になる。

specs/18-sp2-demo-tenancy.md §1（エンティティ・migration 017・状態機械）・§2（CRUD 契約）、specs/17 §6

## 作業内容
- migration `017`〜`021`（specs/18 §1.1 — **1 ファイル = 1 文**。migrate.py のファイル単位記録と
  Oracle DDL 暗黙 commit の制約により複文ファイルは再実行不能になるため）:
  017 = demos への列追加を 1 つの `ALTER TABLE demos ADD (...)` に束ねて原子化
  （description / config・IS JSON CHECK / status・CHECK 4 値 / updated_at）、
  018・019 = demos の索引、020 = conversations.demo_id、021 = その索引（利用は SP2-03）。
- **migrate.py の再実行許容**（specs/18 §1.1 — codex review-8 B001 / review-9 B001）:
  「DDL 成功 → version 記録前クラッシュ」で残る記録なし DDL に対し、既適用示唆の ORA コード
  （-01430/-00955/-01408 等）を検知したら**期待事後条件をデータディクショナリで完全一致検証**
  （USER_TAB_COLUMNS の列・型・DEFAULT / USER_CONSTRAINTS / USER_INDEXES）してから version を
  記録（一致しない同名オブジェクトは停止・人間対応）。fault-injection テスト
  （DDL 直後停止 → 再実行収束・形違いでは停止）を 017〜021 に対して実施。
- `jetuse_core/demos.py` 拡張（specs/18 §1.3。usecases の流儀・SQL で所有者強制）:
  `create_demo(owner, name, description, visibility, config)` / `list_demos(owner)` /
  `update_demo(owner, id, fields)` / `set_status(id, from_status, to_status)`（楽観 WHERE）。
  全 UPDATE 文で `updated_at = SYSTIMESTAMP`。
- routes `service/routes/demos.py` に **CRUD 4 ルート**（specs/18 §2.1 の表のとおり。
  **DELETE は公開しない** — 後始末を持たない行削除は SP1-03 で作られた RAG 箱を孤児化するため、
  DELETE ルートの初公開は後始末込みで SP2-02。specs/18 §2.1）:
  `GET /api/demos`（require_user・自分のみ）、`POST /api/demos`（require_user・INSERT のみ・即 ready）、
  `GET /api/demos/{id}`（require_demo）、`PATCH /api/demos/{id}`（require_demo_owner・status 変更不可・
  null 意味論は specs/18 §2.2）。レスポンスは DemoOut（owner_sub 非公開・mine 付き — specs/18 §2.2）。
- `require_demo` に deleting→404 を追加し、DemoContext に status を追加（specs/18 §2.3）。
- 単体テスト: CRUD 往復・越境 404（存在秘匿の同一形）・public の読み取り可/書き込み不可・401・
  status/owner_sub の変更不可（422 or 無視でなく入力スキーマ非包含）・deleting の 404・
  config 1MB 上限 422・PATCH の null 意味論（空 PATCH / name・visibility・config の明示 null 422 /
  description の null クリア — specs/18 §2.2）・**config.dbchat の形状検証（POST/PATCH 同一契約 —
  dbchat の null/配列/文字列 422・model の null/型違い/未知値 422・省略は既定 — specs/18 §2.2）**。
