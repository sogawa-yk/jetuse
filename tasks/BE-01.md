# タスク: BE-01 デモ起動の実デプロイ配線（launch → OKE 実配備）

## ゴール
ヒアリング `launch` を「DB行＋`/sba/{id}` URL を返すだけ」から、**実 OKE への配備**まで行うよう配線する。
`deploy.py`（宣言的配備仕様の render）/`deploy_inject.py`（base_url ConfigMap＋短期トークン Secret）を
**実 apply 経路**につなぎ、生成デモが OKE 上の独立ワークロードとして起動・削除できるようにする。

## 対象 area
both（api: launch→deploy 配線・ライフサイクルAPI ／ 必要なら web: 配備状態表示）

## 仕様の正
- ADR-0017（OKE 実行基盤）／ADR-0014（秘密は本体のみ・L3へ配らない D5）／ADR-0016（注入ライフサイクル）。
- specs/16-platform.md（Platform API 注入）。親計画 §10 ステージ6 BE-01。
- 現状: `packages/api/jetuse_core/deploy.py` / `deploy_inject.py` は render-only かつ**どの route からも未 import**。
  `jetuse_core/hearing.py`(launch) は DB 行＋URL のみ（コンテナ配備しない）。

## 受け入れ条件（検証可能な述語）
- [ ] `launch` 実行で当該デモが OKE に **Namespace/Deployment/Service** として配備され、Pod が Running・`/healthz` 応答。
- [ ] base_url ConfigMap ＋ 短期トークン Secret が注入され、デモから Platform API ブローカー経由でデータ取得できる（秘密は本体のみ保持）。
- [ ] デモ削除 API で namespace ごと撤去（trivial delete）。display_name 収束・棚卸しが効く。
- [ ] 既存契約（`/sba/{id}` ルーティング・launch レスポンス）を後方互換で維持。
- [ ] `pytest packages/api/tests`（launch/deploy 関連）全パス。lint/型チェッククリーン。

## E2E シナリオ（実環境 / jetuse-dev・最低2本）
- [ ] 正常系: ヒアリング→launch→OKE に Pod 起動→ブローカー注入で実データ取得→画面到達。証跡 `runs/<run-id>/e2e/`。
- [ ] ライフサイクル: 同デモを削除→namespace/ワークロードが消える（残骸なし）。再 launch で再作成。
- [ ] 実 apply を要する範囲で未実施があれば `SKIPPED.md` に明記。

## 非ゴール / 制約
- 新規インフラ基盤の追加（固定 OKE クラスタを使う）。デプロイ上限＝コンテナ。
- 既存リソース（VCN develop / インスタンス dev / 保護バケット）は参照のみ。
- **人間ゲート: terraform apply・課金（OKE 実配備）**。自走では設計＋IaC＋検証まで、実 apply はステージ報告で提示。
- spec-driven: ADR にない判断は実装せず ADR 追補案。
