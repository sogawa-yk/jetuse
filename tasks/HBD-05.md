# タスク: HBD-05 構成サマリ出力（顧客提示用）＋E2E（ヒアリング→デモ起動）

## ゴール
合成・検証済みのデモ構成から、顧客提示用の構成サマリを生成し、
「ヒアリング → Q&A → 合成 → バリデーション → プレビュー → デモ起動」を一気通貫で成立させる（S1+S2 体験の出口判定）。

## 対象 area
both（api：サマリ生成 / web：サマリ表示＋一気通貫導線）

## 依存
HBD-02（ダイアログUI）/ HBD-03（合成・プレビュー）/ HBD-04（バリデーション）。

## 仕様参照
docs/enhance/202607-demo-platform-plan.md §5.1 / §9「S1+S2で体験成立」/ §10「HBD-05」、
docs/enhance/202607-hearing-flow.md §5（推薦構成サマリ＝構成図・使うOCIサービス・デモ手順・想定効果）。

## 受け入れ条件（検証可能な述語で書く）
- [ ] 構成サマリ生成: hearing-flow §5 の項目（①構成図＝どのデータに何のAIが効くか ②使うOCIサービス＝固定リファレンス基盤の該当部分 ③デモ手順 ④想定効果）を出力する
- [ ] サマリ文章化は GenAI 補助だが、構成図・使用サービスは合成結果から**決定的に**導出する（捏造しない）
- [ ] 一気通貫導線: ヒアリング確定→合成→バリデーションPASS→プレビュー→「このデモを起動」で実 loop 環境にデモが立ち上がる
- [ ] サマリは顧客提示資料の下敷きとしてエクスポート可能（プリセールス転用）
- [ ] web build・vitest・eslint / api lint（ruff）クリーン。サマリ生成と一気通貫の結合テストを追加

## E2E シナリオ（実環境 / jetuse-dev・複数）
完了ゲートで loop 環境へデプロイ（DB は専用スキーマ `JETUSE_HBD-05` で隔離）し、証跡を `runs/<run-id>/e2e/` に残す。
- [ ] シナリオ1（一気通貫・正常）: サンプルのヒアリングメモ入力→Q&A確定→合成→バリデーションPASS→デモ起動が成立し、起動デモで主役AI機能が実際に動く（実 GenAI/実 ADB）
- [ ] シナリオ2（顧客提示サマリ）: 起動済みデモの構成サマリ（構成図・使うOCIサービス・手順・効果）が生成・エクスポートされる
- [ ] シナリオ3（境界）: バリデーションFAIL構成では「起動」に進めず代替提案へ誘導される
- [ ] 実施不能な範囲は `runs/<run-id>/e2e/SKIPPED.md` に理由明記

## 成果物
`jetuse_core` のサマリ生成 / web の一気通貫導線＋サマリ画面＋エクスポート / `docs/verification/HBD-05.md`

## 非ゴール / 制約
- コンテナ配備（L3）・マーケット公開は S4。本タスクの「起動」は既存 loop 基盤上のデモ。
- アドバンスドモード（codex背後）・フロント生成は S5。
- **人間ゲート（デモ品質）**: 一気通貫デモの品質を人間が確認してから done。
- 既存リソースは参照のみ。spec-driven。コミット/PR/push は人間承認後。
