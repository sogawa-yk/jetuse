# タスク: SP1-03 デモスコープ能力ルート縦切り（chat + rag）

## 目的
`/api/demos/{demo_id}/...` 配下に能力ルートを敷き、能力が `DemoContext` を受け取って**その箱の中だけ**を
操作する形を確立する（specs/17 §5）。まず chat + rag（ファイル管理）の縦切りで型を作る。
残能力（dbchat/agents/voice/minutes/translate/docunderstand）は同型で追随（SP2 以降）。

## 仕様参照
specs/17-demo-platform-redesign.md §5・§9

## 対象 area
api

## 前提（依存タスク / 人間の事前作業）
SP1-01（カタログ）・SP1-02（DemoContext）。

## 作業内容
- `packages/api/service/routes/demos.py` を新設: `APIRouter(prefix="/api/demos/{demo_id}")`、
  全ルートが `Depends(require_demo)` で `DemoContext` を得る。
  - `POST /api/demos/{demo_id}/chat` — 既存 `routes/chat.py` のストリーミングハンドラを流用し、
    **名前空間キーに `user.subject` の代わりに `ctx.namespace` を渡す**。既存コア関数
    （`stream_chat` 等）のシグネチャは変えない（第一引数のキー文字列を差し替えるだけ）。
  - `GET/POST/DELETE /api/demos/{demo_id}/rag/files` — 既存 `routes/rag.py` と同じ検証
    （拡張子/サイズ/空）で、`rag.list_files(ctx.namespace)` / `rag.add_file(ctx.namespace, ...)` /
    `rag.delete_file(ctx.namespace, ...)` を呼ぶ。デモの箱 = `demo_<id>` 名前空間のベクタストア。
  - ハンドラ本体の重複は避ける: 既存ルートとデモスコープルートが**同じ内部関数を namespace 引数で
    共有**する形にリファクタしてよい（挙動・パス・ステータスは既存と同一を維持）。
- `service/main.py` に router を include。
- SP1-01 のディスクリプタ `chat` / `rag.search` に、デモスコープ側ルート
  （`/api/demos/{demo_id}/chat` 等）を `routes` へ追記する。
- 単体テスト `packages/api/tests/test_demo_routes.py`:
  - 所有者: chat がストリーミング応答を返す（既存 test_chat.py のモック流儀を踏襲）。
  - 所有者: rag ファイル upload → list に現れる → delete。
  - **分離**: ユーザーA のデモに B がアクセス → 404（chat・rag 両方）。
  - **箱の分離**: 同一ユーザーの demo1 に上げたファイルが demo2 の list に現れない。
  - user 単位の既存ルート（`/api/chat` / `/api/rag/files`）の回帰なし（既存テストが緑のまま）。

## 非ゴール / 禁止事項
- dbchat のデモスコープ化はしない（デモ別スキーマ `demo_<id>` のプロビジョニングが SP2 のため）。
- デモの静的SPA配信・ビルダーは対象外（SP3）。
- 既存 user 単位ルートのパス・挙動・ステータスを変えない。認証情報のコミット禁止。
