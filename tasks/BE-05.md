# タスク: BE-05 スコープ承認 API＋UI（PAPI-02 の到達経路）

## ゴール
実装済みだが到達経路の無い `platform_grants.approve_scopes`（PAPI-02）に **REST ルート＋承認 UI** を足し、
manifest 境界内でブローカートークンに scope を付与できるようにする。これにより `/api/platform/*` が実運用で通る。

## 対象 area
both（api: 承認/取消ルート ／ web: スコープ承認 UI）

## 仕様の正
- specs/16-platform.md §13.3/§13.4（スコープ承認・発行）。ADR-0014（スコープ語彙・テナント境界・監査）。
- 親計画 §10 ステージ6 BE-05。
- 現状: `platform_grants.approve_scopes` は実（`MERGE INTO platform_scope_grants`・manifest 境界）だが **REST ルートが無い**ため、
  ブローカートークンに scope を付与する経路が UI/API に無い。

## 受け入れ条件（検証可能な述語）
- [ ] 承認/取消の REST ルート（例 `POST/DELETE /api/platform/grants`）を追加。manifest 宣言スコープの範囲内のみ許可（二重閉包）。
- [ ] 監査ログ（誰が・いつ・どの scope を承認）。fail-closed（範囲外 scope は拒否）。
- [ ] web に承認 UI（プロジェクト×scope のレビュー→承認）。
- [ ] 承認後に発行されるブローカートークンが当該 scope を保持し `/platform/*`（db/connector/rag）が通る。
- [ ] `pytest`（platform_grants / platform route）＋ web テスト全パス。lint/型/`npm run build` クリーン。

## E2E シナリオ（実環境 / jetuse-dev・最低2本）
- [ ] 正常系: UI で scope 承認→トークン発行→`/api/platform/db/query` が当該 scope で成功。証跡 `runs/<run-id>/e2e/`。
- [ ] 境界系: manifest 宣言外 scope の承認を拒否（fail-closed）＋監査記録。

## 非ゴール / 制約
- 新しいスコープ語彙の追加（既存 §4 集合を使う）。
- 承認は人間操作（自動承認しない）。spec 外は ADR 案。既存リソース参照のみ。
