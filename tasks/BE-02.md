# タスク: BE-02 サンプルアプリのデータ自動マテリアライズ（NL2SQL を実データで）

## ゴール
デモ起動/scaffold 時に、サンプルアプリの dataset を **実テーブルとして自動生成・シード投入し、読取ユーザー
（`adb_query_user`=JETUSE_QUERY）へ SELECT 付与**する。これにより SBA-B 在庫照会等の NL2SQL が
**新規デモ起動だけで動く**（手動マテリアライズ不要）。`SAMPLE_DB_SCHEMA` と CURRENT_SCHEMA 解決を整合させる。

## 対象 area
api（jetuse_core: synth / scaffold / datasets / nl2sql 連携）

## 仕様の正
- 親計画 §10 ステージ6 BE-02。memory [[jetuse-oke-deploy-feature-gaps]]（②本質ギャップ）。
- 現状: `jetuse_core/synth.py:221` の SeedPlan は **plan-only**（行数を数えるだけ・INSERT しない）。
  実テーブル生成＋seed＋GRANT は ENH-01 の CSV 経路 `jetuse_core/datasets.py:207-218` にしか無く、デモ compose/launch から呼ばれない。
  `nl2sql.execute_readonly(current_schema=...)` は `SAMPLE_DB_SCHEMA` 設定時に CURRENT_SCHEMA を固定する。

## 受け入れ条件（検証可能な述語）
- [ ] デモ起動（または scaffold）で dataset 定義から **実テーブルを CREATE＋seed 投入**し、`adb_query_user` へ `GRANT SELECT`。
- [ ] 冪等（再起動で重複作成しない／既存は再利用 or 安全に再作成）。スキーマは `adb_user`(JETUSE_APP)、読取は CURRENT_SCHEMA 固定で解決。
- [ ] SBA-B（inventory/orders）で **新規デモ起動のみ**で NL2SQL invoke→execute が成功（ORA-00942 が出ない）。
- [ ] 列型（string/number/date）を適切な Oracle 型へマップ。生成 SQL ガード（許可テーブル内）は維持。
- [ ] `pytest packages/api/tests/test_synth.py` ＋新規マテリアライズ/権限テスト全パス。lint/型クリーン。

## E2E シナリオ（実環境 / jetuse-dev・最低2本）
- [ ] 正常系: 新規 SBA-B デモ起動→inventory/orders 実テーブル生成・GRANT→「カテゴリ別在庫数」NL2SQL が実行され結果表示。証跡 `runs/<run-id>/e2e/`。
- [ ] 権限/分離: 読取ユーザーが許可テーブルのみ参照（他スキーマ/辞書ビュー拒否）。別デモのデータと混ざらない。
- [ ] loop ADB（jetuse-loop-adb）再利用・専用スキーマ隔離で実施。

## 非ゴール / 制約
- ユーザーCSVアップロード(dbchat)経路の改変（既存 `datasets.py` は再利用に留める）。
- 大規模データ生成（デモ用シード規模）。
- **人間ゲート: 実 ADB 利用（loop ADB 再利用可）**。既存リソースは参照のみ。
- spec-driven: 仕様外判断は ADR 案。
