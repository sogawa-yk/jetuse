spec-driven の原則（CLAUDE.md）どおり、**仕様にない実装判断を実装前に人間承認へ**通す。

## 対象 area

## 前提（依存タスク / 人間の事前作業）
なし（第0波・単独）。

## 作業内容
- `specs/19-sp3-builder.md` を起草する。最低限決めること:
  - **ビルダーのパイプライン全体**: ヒアリング → デモ設計 → フロント生成 → データ生成 → `Demo` 保存 の
    各段のフェーズ境界・状態遷移・成果物（中間アーティファクト）と、失敗時の巻き戻し。
  - **ヒアリング(NL) の契約**: セッション状態の持ち方（会話 or 専用エンティティ）、収集する項目
    （業種/ユースケース/使う能力/データの雰囲気）、いつ「設計に足りる」と判定するか。
  - **デモ設計の契約**: 能力カタログ（`GET /api/capabilities` の `demo_safe=true`）を LLM に渡し、
    **デモプラン**（採用能力・画面構成・各能力の配線・必要データのスキーマ）を機械可読な成果物として産出。
    プランのスキーマ（バリデーション可能な形）を定義する。
  - **フロント生成方式（OpenCode + OCI モデル）**: どう静的SPA を生成するか。**サーバ側で OpenCode を
    どのランタイムで回すか・生成コードのサンドボックス・生成物が JetUse API のデモスコープしか叩けない
    ことの構造的担保・秘密を埋め込ませない**。ここは技術リスクが高いので、**実装方式は SP3-03 の ADR で
    確定**する前提（specs/19 では要件と受け入れ条件・非機能条件を定義、方式選定は ADR に委譲）。
  - **生成デモの保存/配信**: 生成した静的SPA バンドルの保管先（デモごと・object storage 想定）と、
    `/api/demos/{id}/...` での配信（specs/17 §1・§5）。既存 `Demo.config` との関係。
  - **サンプルデータ生成/投入**: デモプランのデータスキーマから `demo_<id>` スキーマ（SP2-02 の
    プロビジョニング機構）へ表作成＋データ投入。dbchat がそのデータを引ける形。
  - **SP3-01〜05 の受け入れ条件の確定**（tasks/SP3-0X.md と STAGE3-PROGRESS.md の更新も本タスクの成果物）。
- 判断が割れる論点（例: フロント生成を同期/非同期・ジョブ化するか / OpenCode をコンテナ内 subprocess か
  別ワーカーか / 生成物のサンドボックス方式）は選択肢と推奨を明記し、必要なら `docs/comparison/` に
  比較ドキュメントを添える（比較ドキュメント主義）。**方式決定を伴うものは ADR ドラフトへ回し SP3-03 で承認**。
