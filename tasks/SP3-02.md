# タスク: SP3-02 デモ設計（能力カタログ→デモプラン生成）

> 受け入れ条件は **specs/19 §3（+ §9 SP3-02）で確定**(SP3-00 起草済み。
> **specs/19 の人間承認が着手条件**)。

## 目的
ヒアリングの要求サマリ（SP3-01）と能力カタログ（SP1 `GET /api/capabilities`）を LLM に渡し、
**デモプラン**を機械可読・バリデーション可能な成果物として産出する。プランが以降の生成
（SP3-03/04）の唯一の入力になる。

## 仕様参照
specs/19-sp3-builder.md §3（設計契約・プランスキーマ・語彙）、specs/17 §3（能力カタログ）・§4・§5。

## 対象 area
api

## 前提（依存タスク / 人間の事前作業）
SP3-01（ヒアリング要求サマリ）。

## 作業内容（specs/19 で確定）
- `POST /api/builder/sessions/{sid}/design`（specs/19 §3.1 — sufficient=true 前提 409・
  検証不合格は同一リクエスト内で最大 2 回再生成・designed 後の上書き再実行可）。
- デモプラン plan_version=1 スキーマ（specs/19 §3.2 — 採用能力・画面/ブロック構成・データ定義。
  **配線はブロック型で固定**し URL/パスの自由記述フィールドを持たない）。
- 検証 fail-closed（specs/19 §3.3 — pydantic strict/extra=forbid・語彙部分集合・識別子/型の
  許可リスト・件数/長さ上限・能力⇔データ定義の整合・直列化 ≤256KB）。
- **プラン語彙の構造的導出**（specs/19 §3.4 — `demo_safe=true` かつデモスコープルートを持つ能力
  = 現状 chat / rag.search / dbchat。ハードコードでなくカタログから導出）。
- 単体テスト: 語彙外能力 422・語彙導出・識別子/型/上限の境界・データ整合違反・再試行で合格・
  スナップショット（固定 fake 出力 → 検証済みプラン）。

## 完了条件（検証可能な述語で — specs/19 §9 SP3-02 と同一）
- [ ] 要求サマリ＋実カタログから**スキーマ検証済みデモプラン**を産出する。
- [ ] プランは specs/19 §3.4 の語彙のみで構成される（範囲外・未知能力を構造的に排除。
  導出はカタログの routes 実在から）。単体テスト全緑・ruff クリーン・回帰なし。
- [ ] 実環境 E2E（プレビュー環境の実カタログ・実 LLM）: 要求サマリ → 検証合格プラン産出を
  `runs/<run-id>/e2e/` に記録。

## E2E シナリオ
- プレビュー環境の `GET /api/capabilities` 実データ＋実 GenAI で design 実行 → 検証合格プランを実機確認。

## 成果物
- コード（デモ設計・プランスキーマ・検証・テスト）・`docs/verification/SP3-02.md`。

## 非ゴール / 禁止事項
- フロント生成（SP3-03）・データ実投入（SP3-04）。カタログ出力形の変更（SP1 の契約は不変）。
- プラン語彙の拡張（stateless 系のデモスコープ化 — specs/19 §11）。認証情報のコミット禁止。
