# タスク: SP3-02 デモ設計（能力カタログ→デモプラン生成）

> 受け入れ条件は specs/19 準拠で確定（SP3-00 起草。**specs/19 の人間承認が着手条件**）。以下は暫定。

## 目的
ヒアリングの要求サマリ（SP3-01）と能力カタログ（SP1 `GET /api/capabilities` の `demo_safe=true`）を
LLM に渡し、**デモプラン**を機械可読な成果物として産出する。プランが以降の生成（SP3-03/04）の入力になる。

## 仕様参照
specs/19-sp3-builder.md（デモ設計・デモプランのスキーマ）、specs/17 §3（能力カタログ）・§4（公開能力）・§5。

## 対象 area
api

## 前提（依存タスク / 人間の事前作業）
SP3-01（ヒアリング要求サマリ）。

## 作業内容（specs/19 で確定・暫定）
- 能力カタログを取得し、要求サマリと合わせて LLM でデモを設計する。産出 = **デモプラン**:
  採用能力（カタログの capability 集合）・画面/導線構成・各能力の配線（どの画面がどの能力を叩くか）・
  必要データのスキーマ（表・列・件数感）。
- デモプランは**バリデーション可能なスキーマ**で表現し、`demo_safe=true` 以外の能力や未知能力を
  含まないことを検証（fail-closed）。プランは `Demo.config` へ載る形と整合させる（SP2 の config 契約）。
- 単体テスト: カタログにない能力を選ばない・プランのスキーマ検証・要求サマリ→プランの決定性
  （温度固定 or スナップショット）・危険/範囲外能力の除外。

## 完了条件（検証可能な述語で）
- [ ] 要求サマリ＋カタログから**スキーマ検証済みデモプラン**を産出する。
- [ ] プランは `demo_safe` 能力のみで構成される（範囲外・未知能力を構造的に排除）。単体テスト全緑・ruff クリーン。
- [ ] 実環境 E2E（デプロイ済み環境の実カタログ・実 LLM）: 要求サマリ→デモプラン産出を `runs/<run-id>/e2e/` に記録。

## E2E シナリオ
- デプロイ済み環境の `GET /api/capabilities` 実データ＋実 GenAI でプラン生成を実機確認。

## 成果物
- コード（デモ設計・プランスキーマ・テスト）・`docs/verification/SP3-02.md`。

## 非ゴール / 禁止事項
- フロント生成（SP3-03）・データ実投入（SP3-04）。カタログ出力形の変更（SP1 の契約は不変）。認証情報のコミット禁止。
