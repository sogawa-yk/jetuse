# タスク: SP3-09 共有テナンシ認証情報の OCI Vault 化

## 目的
OpenCode バックエンド（gpt-5 系・ORASEJAPAN 共有テナンシ）の API 署名鍵材料を、
RM スタックの sensitive 変数から **OCI Vault（jetuse:dev）のシークレット**へ移す
（施主指示 2026-07-09）。デプロイされた API はリソースプリンシパルで Vault から取得する。

## 背景（確定事実）
- 現行（SP3-07）: 鍵材料 = RM 変数 GEN_SHARED_{KEY_PEM_B64,USER_OCID,TENANCY_OCID,FINGERPRINT}
  （sensitive）→ entrypoint が起動時に ~/.oci を umask 077 で生成（fail-closed・ログ非出力）。
- jetuse:dev コンパートメントに ACTIVE な Vault は無い（2026-07-09 確認）→ 新規作成
  （**software 保護キーで作る** — 追加課金なし。Terraform 管理・infra/terraform/environments/dev-app
  か隣接モジュール。KMS Vault は削除に猶予期間がある点に注意 = むやみに作り直さない）。
- API CI の RP は DG jetuse-internal-dg（manage all-resources in jetuse:dev）で secret-family 読取可
  （IAM 変更不要の想定。不可なら止めて報告 — IAM は人間ゲート）。
- ローカル（dev インスタンス / podman ランタイム）は ~/.oci の ORASEJAPAN プロファイルを継続使用
  （Vault は**デプロイ環境の経路**。設定の優先順位を明文化）。

## 対象 area
api（起動時/遅延の secret 取得・sign_proxy への供給）+ infra（Vault/Key/Secret の tf・RM 変数整理）

## 作業内容
1. **Vault/Key/Secret の IaC**: jetuse:dev に Vault（software key）+ シークレット
   （鍵材料一式 — 形式は 1 つの JSON シークレット推奨: user/tenancy/fingerprint/region/key_pem。
   実値は tf に書かず、`ops/deploy-dev-app.sh seed-env`（or seed-vault）がローカル ~/.oci の
   ORASEJAPAN プロファイルから読んで CLI で投入・更新する）。
2. **API 側**: デプロイ環境では RP の secrets クライアントで取得して署名材料を構成
   （環境変数に鍵を経由させない。in-memory が理想 — oci-genai-auth がファイル前提なら
   umask 077 の一時ファイル生成でも可）。取得失敗は現行同様 **fail-closed**（共有テナンシ
   モデルのみ無効・API は起動継続・値はログに出さない）。設定は GEN_SHARED_SECRET_OCID
   （非鍵材料）1 個へ集約。
3. **RM 変数の掃除**: 旧 GEN_SHARED_KEY_PEM_B64 等の鍵材料変数をスタックから除去し、
   除去されたことを stack get で確認（証跡）。seed-env の該当部分を Vault 投入へ置換。
4. docs 更新（dev-environments.md の秘匿値の流れ・ローテーション手順 = シークレット新版投入 →
   再デプロイ/再起動）。

## 非ゴール / 禁止事項
- ローカル開発経路（~/.oci プロファイル）の変更。IAM/ポリシー/DG の変更（不足時は停止して報告）。
- Vault の乱造（1 Vault を再利用。destroy しない）。他の秘匿値（ADB パスワード等）の Vault 化は
  対象外（余力があっても手を広げない — 必要なら residual に提案として記載）。
- 認証情報・OCID 実値のコミット禁止。共有スタックの destroy 禁止。
