# タスク: SP2-03 DemoContext 解決先の実装 + dbchat デモスコープ化

> 受け入れ条件は specs/18 §4 準拠で確定（SP2-00 起草。**specs/18 の人間承認が着手条件**）。

## 目的
`DemoContext` の解決先を完成させ（会話の demo_id 紐付け — SP1-03 の 422 暫定拒否を解除）、
SP1-03 で非ゴールだった **dbchat のデモスコープ化**（`/api/demos/{demo_id}/dbchat|db/...`）を
SP1-03 の rag と同型で敷く。これで主要 3 系統（chat / rag / dbchat）が縦に揃う。

## 仕様参照
specs/18-sp2-demo-tenancy.md §4（DemoContext・会話紐付け・dbchat 契約）、specs/17 §5・§6、
SP1-03（同型の先例: 共有内部関数化）

## 対象 area
api

## 前提（依存タスク / 人間の事前作業）
SP2-02（deleting ゲート込みの箱ライフサイクル。migration 020 の conversations.demo_id は SP2-01 で適用済み）。

## 作業内容
- 会話のデモ紐付け（specs/18 §4.2）: demo chat の conversation_id 422 拒否を解除。
  `POST /api/demos/{id}/conversations`（require_demo）を追加し demo_id 付きで作成
  （既存 POST /api/conversations と同形レスポンス。chat の SSE は会話を自動作成しない）。
  chat での指定時は `owner_sub = owner_key(user.subject) AND demo_id = ctx.demo_id` を検証
  （不一致 404。**資源キー列は全経路で owner_key ヘルパー経由** — raw identity 列である
  demos.owner_sub とは区別する。specs/18 §4.2。予約接頭辞 sub の往復テストを含む）。
  **demo 会話は OCI Conversation を作らない**（specs/18 §4.2 — 外部会話の後始末・クラッシュ孤児・
  LTM のデモ間混線を構造的に排除。demo chat の LTM は無効）。継続契約 = **クライアントが
  messages に全履歴を再送**（既存 SPA と同じ流儀。conversation_id は保存と箱の紐付けのみ）。
  **user 単位の会話 API は全 verb（一覧/GET/DELETE/title/chat 継続・履歴保存）で
  `demo_id IS NULL` を強制**（両方向の持ち込みが 404。既存データは全行 NULL のため
  挙動不変 = 回帰なしをテストで担保）。
- dbchat 縦切り（specs/18 §4.3 の表のとおり。既存ハンドラを namespace 引数の共有内部関数へ整理）:
  nl2sql（datasets ターゲット固定・ensure_profile(namespace)）/ execute / schema、
  および datasets CRUD（list/create/generate/preview/delete）のデモスコープ版。
  読み取り = require_demo、書き込み = require_demo_owner。
- **datasets 表の越境防止**（specs/18 §4.3。**VPD 基盤〔DBMS_RLS 確認・DB オブジェクト・
  人間承認・完全性検証・set/clear〕は SP2-02 が所有** — 本タスクはその基盤の上に重ねる）:
  ① **層2 = fail-closed の SQL ゲート**を `execute_readonly` に実装（呼び出し元 owner キー必須。
  拒否対象: 登録簿外の `JETUSE_DS_` 参照 403 / **データディクショナリ・動的ビュー**
  （ALL_/DBA_/USER_/CDB_/V$/GV$ — VPD は辞書のメタデータを隠さないため、他 owner の表名・
  列名の取得をここで塞ぐ）/ DBMS_/UTL_/SYS. 等のパッケージ呼び出し / `@` DB リンク・synonym。
  判定不能は拒否側に倒す。SH 許可リストと一般 SQL 関数は許可）。
  ② **全呼び出し元**（user/デモ execute・preview 系・datasets.preview・`query_database`
  ツール・Fn ルーター）の移行契約と SH 等固定スキーマ経路の分離は specs/18 §4.3。
  owner キーは導出ヘルパー経由に統一。**agent 経路**（`tools.py` / agent-containers
  `agent_db.py` — execute_readonly を通らない独立経路）にも層2ゲートを **owner なしモード**で
  適用（JETUSE_DS_/辞書/パッケージ全拒否 — データ行は VPD fail-closed が遮断。owner_key の
  invoke 伝播は SP3 課題。specs/18 §4.3）。main への展開は別 PR（residual）。
  ③ **demo nl2sql のモデルは demo config 固定**（非所有者のモデル指定でプロファイル再構築を
  起こさせない — 変更は owner の PATCH のみ。specs/18 §4.3）。
- capabilities の `dbchat` ディスクリプタにデモスコープルートを追記（routes 実在テストが検証）。
- 単体テスト: 所有者の NL2SQL 往復（モック）・越境 404・越境防止
  （**user→demo / demo→user / demo A→demo B / user A→user B の 4 方向 403**、
  本人データセット + SH は従来どおり 200、`execute_readonly` 全呼び出し元の回帰、
  辞書ビュー/パッケージ/synonym 参照の 403、非所有者のモデル指定が無効）・
  **public デモの非所有者**（nl2sql/execute/schema/list/preview が demo namespace で 200、
  create/generate/delete は同一形 404 — require_demo 契約の回帰検出）・
  他人/他デモの conversation_id 404・user 経路との両方向持ち込み 404・
  **demo chat が OCI Conversation を作らない/LTM がデモ間で混ざらない**（fake）・
  user 単位 dbchat / conversations 全 verb の回帰なし・箱の分離。
