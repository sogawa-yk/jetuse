# タスク: SP3-08 生成ランタイムの実 Container Instance 化（jetuse:dev で生成 E2E 可能に）

## 目的
フロント生成の実行環境を host podman 近似から **ADR-0023 §1 の本来設計 = 生成ごとの使い捨て
OCI Container Instance** へ移行し、**jetuse:dev にデプロイされた API 上でビルダーの「生成」が
end-to-end で動く**ようにする（施主指示 2026-07-09: 以降の E2E は jetuse:dev のリソースで行う。
ADR-0023「PoC 受理と descope 記録」の descope 項目②の前倒し実装）。

## 背景（確定事実）
- 現行 runtime（jetuse_core/generation_runtime.py）は host podman の 2 相（生成相/信頼ビルド相）。
  デプロイ先の Container Instance 内に podman は無い = デプロイ環境で generate は動かない。
- ADR-0023 §1: ランタイム B' = 相ごとの使い捨て CI（CI.Standard.E4.Flex 1ocpu/4GB・起動実測 43 秒・
  命名 `jetuse-builder-{gen,build}-<job_id>`・reconcile が命名から回収）。API の RP は
  DG jetuse-internal-dg（manage all-resources in jetuse:dev）で CI create/delete 可能（SP2/ADR 実測）。
- PoC 緩和（施主承認 2026-07-08）: NSG 完全隔離・PID/ディスク上限強制は不要。ただし
  S2（生成コンテナに資格情報を渡さない）と S3/S4/S5（静的検査・原子的公開）は維持する。
- 生成イメージ（OpenCode + vendored scaffold 同梱）は SP3-03/06 でビルド済みの系譜あり（OCIR）。
- 署名プロキシ（sign_proxy.py）はデプロイ側で稼働している前提（SP3-07 が配備方式を確定）。
  生成 CI からプロキシへは VCN 内到達（プライベート IP/サブネット — 到達方式は実機で確定）。

## 対象 area
api（generation_runtime の CI バックエンド・reconcile）+ infra（必要な tf/変数）

## 作業内容
1. **CI ランタイム実装**: `build_frontend` シームの Container Instance 実装を追加
   （設定で podman / oci-ci を切替 — ローカル開発と単体テストは podman/モックのまま）。
   - 相1: 生成 CI 起動（プラン・スキャフォールドの受け渡し）→ OpenCode 実行 → src 回収 → CI 削除。
   - 相2: 信頼ビルド CI で検査+build → dist 回収 → CI 削除。既存の静的検査・原子的公開は不変。
   - **成果物受け渡しは鍵レスを維持**: 推奨 = ジョブ専用 Object Storage prefix + **期限付き PAR**
     （書込専用/読取専用を相・方向ごとに分離。CI に OCI 資格情報や RP を与えない —
     `is_resource_principal_disabled=true`）。別方式が実機上優位ならそれでよい（ADR に追記）。
   - タイムアウト・15 分ハードキャップ・失敗時 failed + CI 補償削除（reconcile: 起動時+定期、
     `jetuse-builder-*` 命名から exact 回収）は ADR §1/§4 どおり。
2. **プロキシ到達**: 生成 CI → 署名プロキシの URL を環境から解決（SP3-07 の配備方式に整合）。
3. **infra**: 生成 CI 用サブネット/セキュリティ設定・イメージ参照など必要最小限を dev-app tf へ
   （SP3-07 の変更とコンフリクトしないよう、可能なら別 tf ファイルに分離）。
4. **E2E（本丸）**: **jetuse:dev にデプロイされた API**（SP3-07 のパイプライン成果 or 同一手順の
   ops スクリプトでデプロイ）に対し、ビルダー UI/API から「プラン → generate → 実 CI 2 相 →
   ready → /app/ 配信 → 生成デモ chat 実応答 → DELETE で後始末（CI 残骸ゼロ含む）」を
   gpt-oss-120b と gpt-5 系 1 モデル（推奨 gpt-5.6-sol）で実施し runs/<run-id>/e2e/ に記録。
5. docs: ADR-0023 へ「実 CI 化完了（descope 解消）」を追記・docs/verification/SP3-08.md・
   N1（生成時間）の実 CI 分布を実測して ADR §5 の推定値を確定値へ更新。

## 完了条件（検証可能な述語で）
- [ ] デプロイ環境（jetuse:dev）でビルダーの生成が実 CI 2 相で完走し、生成デモが動く（上記 E2E）。
- [ ] 生成 CI に資格情報ゼロ（RP 無効・PAR 等の期限付き受け渡し）を実機確認（負の確認込み）。
- [ ] 生成失敗/タイムアウト時に demo=failed + CI が残らない（reconcile 実機確認）。
- [ ] 単体テスト（CI バックエンドはモック）・pytest/ruff 全緑・podman 経路の回帰なし。
- [ ] review_verdict=PASS。技術限界に当たったら止めて findings を記録し判断を仰ぐ（churn 禁止）。

## E2E シナリオ
- jetuse:dev デプロイ環境で新規デモ 1 件を生成し切る（120b と gpt-5.6-sol の 2 本）→ 削除まで。
- 生成中の CI が `jetuse-builder-*-<job_id>` で観測でき、完了/失敗後に残骸ゼロ。

## 非ゴール / 禁止事項
- 本番堅牢化 major 群の全消化（ジョブトークン状態機械のフル実装等 — PoC 範囲の受け渡しでよい）。
- NSG 完全隔離・PID/ディスク上限強制（PoC 緩和のまま）。IAM/DG の変更（人間）。
- 共有スタックの destroy。認証情報・OCID 実値のコミット禁止。SP3-07 と同一ファイルの大改変
  （コンフリクト最小化 — 直列実行だが配慮する）。
