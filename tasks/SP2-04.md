# タスク: SP2-04 Internal テナンシ分離（Identity Domains 実接続）

> 受け入れ条件は specs/18 §5 準拠で確定（SP2-00 起草。**specs/18 の人間承認が着手条件**）。
> **人間の事前作業が必須**（specs/18 §5.2 の一覧）。未完なら stage-runner は本タスクを blocked にして先へ進む。

## 目的
Internal 版（施主ホスト・SA がアクセス）のユーザー分離を実 Identity Domains で確立する。
SP1 の E2E で SKIPPED になっていた「実 IdP トークンでの検証」を解消し、
Internal 配備では `AUTH_REQUIRED=true` を既定にする。

## 仕様参照
specs/18-sp2-demo-tenancy.md §5（設定契約・既定値・人間側作業）、specs/17 §1

## 対象 area
api（＋Internal 配備設定）

## 前提（依存タスク / 人間の事前作業）
- SP2-00（specs/18 承認）。SP2-01〜03 とは独立に進められる。
- **人間の事前作業（specs/18 §5.2 の表。IAM / Identity Domain は人間ゲート — CLAUDE.md）**:
  1. Identity Domain への JetUse 用アプリ登録 → issuer / audience / JWKS URL / client_id の提供。
  2. テストユーザー 2 名の払い出し + 実トークンの取得手段（越境 404 E2E 用）。
  3. Internal 配備先 secret / env への実値投入。

## 作業内容
- 実 IdP の OIDC 設定を `.env` 契約（`OIDC_ISSUER` / `OIDC_AUDIENCE` / `OIDC_JWKS_URL` —
  settings.py 実装済み）で受けて検証経路を実機確認。コードは SP1-03 でローカル JWKS の
  RS256 検証済みのため、**実 Identity Domains 固有の claim 差分のみ吸収**。
  **issuer は固定値を前提にしない**（specs/18 §5.1 — 世代で契約が異なるため、人間から受領した
  discovery の issuer をそのまま設定する）。
- `.env.example` に `OIDC_AUDIENCE` 行を追記し、既存の固定 issuer 行をプレースホルダへ改める
  （実値はコミットしない。issuer/JWKS は discovery、audience はアプリ登録の primary audience
  または実トークンの `aud` 実測 — specs/18 §5.1）。
- **auth.py の fail-closed 強化（specs/18 §5.1 — codex review-2 B006）**: `AUTH_REQUIRED=true` の
  とき issuer / audience / JWKS URL の**3 つ全てを必須**（不備は 500 — 検証を欠いたまま受理しない）。
  検証後の `sub` が欠落/空なら 401。wrong issuer / wrong audience / missing sub の拒否テストを追加。
- Internal（dev 枝）の配備設定（OKE manifest / ORM スタック env）で `AUTH_REQUIRED=true` を既定化。
  **コード上の `Settings.auth_required=False` は変えない**（Public/main の既定を壊さない）。
- 表示名が必要な画面のみ claims の `user_displayname` → 無ければ sub（specs/18 §5.1。
  ユーザーテーブルは作らない）。
