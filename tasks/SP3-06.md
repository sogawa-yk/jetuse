# タスク: SP3-06 生成モデルの UI 選択（gpt-5 系・ORASEJAPAN テナンシ対応）

## 目的
ビルダーの**フロント生成（OpenCode バックエンド）の LLM を UI から選択可能**にする。
選択肢は既定 gpt-oss-120b（自テナンシ大阪）+ 施主指定の 7 モデル（ORASEJAPAN テナンシ）:
gpt-5.1-codex-mini / gpt-5.3-codex / gpt-5.5 / gpt-5.5-pro / gpt-5.6-luna / gpt-5.6-sol / gpt-5.6-terra。
（施主指示 2026-07-08。生成 UI 品質向上の第二レバー — ADR-0023 F2 の拡張）

## 事前調査で確定済みの事実（orchestrator probe 2026-07-08。再検証不要・前提としてよい）
- OCI CLI プロファイル **ORASEJAPAN**（~/.oci/config 設定済み・region タイポ修正済み）で
  SE Japan 共有テナンシへアクセス可。**認可は個人コンパートメント `yuki.sogawa` 経由必須**
  （テナンシ root は 404。OCID は iam compartment list --access-level ACCESSIBLE で解決し .env へ）。
- OpenAI 互換エンドポイントの疎通結果（compartment はヘッダ CompartmentId / opc-compartment-id）:
  - `chat/completions` **OK**: gpt-5.5（ap-osaka-1）、gpt-5.6-luna / sol / terra（us-chicago-1）
  - `chat/completions` **404**（"not supported... use responses"）→ **`responses` で OK**:
    gpt-5.1-codex-mini / gpt-5.3-codex / gpt-5.5-pro（いずれも ap-osaka-1）
- `oci-genai-auth` の `OciUserPrincipalAuth(profile_name=...)` はプロファイル指定対応済み。

## 仕様参照
specs/19 §4.1 F2（モデル切替）・§4.5（generate API — model パラメータの追補を本タスクで行う）、
ADR-0023 §2（署名プロキシ・allowlist）・§6（モデル実測）。

## 対象 area
api（プロキシ・生成オーケストレーション）+ web（ビルダー UI のモデル選択）

## 作業内容
1. **署名プロキシの多目的化**（spikes/sp3_03_sign_proxy.py を昇格 or 後継モジュール）:
   - **生成モデルルーティング表**: モデル → {region エンドポイント, auth プロファイル,
     compartment, api 種別(chat|responses)}。環境依存値（プロファイル名・compartment OCID）は
     .env、表の構造はコード。既定 gpt-oss-120b は現行どおり自テナンシ（RP/DEFAULT）。
   - 許可パスを `chat/completions` + `responses` の **2 つの完全一致**に拡張（他は 403 のまま）。
     本文サイズ上限・ヘッダ固定・SSE 素通し等の既存防御は維持。
   - モデル allowlist を上記 8 モデルに拡張。
2. **生成モデルレジストリの分離**: 共用 MODELS に gpt-5 系を**足さない**（チャット UI へ漏れて
   自テナンシで 404 になるため）。生成専用レジストリ（key → oci_id / api / 表示ラベル）を新設し、
   `generation_model` 設定と generate API の検証をそちらに向ける。
3. **generate API に model パラメータ**: `POST /api/builder/sessions/{sid}/generate` body に
   任意 `{"model": <生成レジストリ key>}`（未指定は設定既定）。検証 fail-closed（未知キー 422）。
   `config.frontend.generator.model` への記録は既存（N6）。specs/19 §4.5 に追補を書く。
4. **OpenCode の responses 対応**（codex/pro 系 3 モデル用・技術リスク）: 現行 provider は
   `@ai-sdk/openai-compatible`（chat/completions のみ）。`@ai-sdk/openai` provider 等で
   baseURL=プロキシの responses 経路を話せるか**最初にスパイク**（jetuse-spike- 規律）。
   成立しなければ該当 3 モデルを UI で選択不可（disabled + 理由表示）にして descope し、
   findings を記録（施主 PoC-first 方針 — churn しない）。
5. **ビルダー UI**: 生成開始 UI にモデル選択（既定 gpt-oss-120b。ラベルに速度/品質の目安、
   i18n ja/en）。選択値を generate body で送る。localStorage 復帰対象。
6. **usage/N5**: 生成 usage 記録の流儀は既存のまま（owner 紐づけ）。ORASEJAPAN 側課金で
   ある旨を docs（tips か verification）に明記。

## 完了条件（検証可能な述語で）
- [ ] UI のモデル選択 → generate → `config.frontend.generator.model` に反映、が通る。
- [ ] プロキシ: 非 allowlist モデル 403・許可 2 パス以外 403・codex 系は responses へ、
  chat 系は chat/completions へルーティングされる契約テスト。
- [ ] 生成レジストリ外キーの generate は 422。共用 MODELS（チャット側）に gpt-5 系が現れない。
- [ ] pytest / ruff / web テスト / lint / build 全緑。回帰なし。
- [ ] 実環境 E2E（host 環境 or プレビュー）: **少なくとも chat 系 1 モデル（gpt-5.6-sol 推奨）と
  responses 系 1 モデル（gpt-5.1-codex-mini 推奨）**で「プラン → 生成 → ready → 配信 → 生成デモ
  実応答」を確認し runs/<run-id>/e2e/ に記録（7 モデル全ての実生成は不要 — 疎通は probe 済み。
  コスト配慮）。gpt-oss-120b の既定経路が無変更で通る回帰確認も 1 本。
- [ ] responses 対応が不成立の場合: 該当モデル disabled + findings 記録で PASS 可（descope 明記）。
