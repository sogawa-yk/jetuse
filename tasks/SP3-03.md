# タスク: SP3-03 フロント生成（OpenCode + OCI モデル→静的SPAバンドル）+ デモ配信

> 受け入れ条件は **specs/19 §4・§5（+ §9 SP3-03）で確定**(SP3-00 起草済み。
> **specs/19 の人間承認が着手条件**)。
> **本タスクは技術リスクの核**。実装前に **ADR（OpenCode 統合方式 — specs/19 §4.4 の 6 項目）を
> 起草→人間承認**（adr_approval ゲート）。予備比較: docs/comparison/frontend-generation-runtime.md。

## 目的
デモプラン（SP3-02）から、OpenCode + OCI モデルで**静的SPA バンドル**を生成し、デモごとに保管して
`/api/demos/{id}/app/` で配信する。生成フロントは JetUse API をデモスコープで叩くだけ（バックエンド生成なし）。

## 仕様参照
specs/19-sp3-builder.md §4（機能 F1〜F6・非機能 N1〜N7・安全 S1〜S6）・§4.5（生成オーケストレーション）・
§5（保管・配信・config 正規キー・後始末 3g）・§8.2（リース整合）、specs/17 §1・§5・§6。
**ADR（本タスクで起草）**: OpenCode 統合方式（specs/19 §4.4 — ランタイム・LLM 認証経路・
egress/オフラインビルド・非同期実行体・上限実測・OpenCode 実挙動）。

## 対象 area
api（生成オーケストレーション）。生成物は静的SPA（人手では書かない）。

## 前提（依存タスク / 人間の事前作業）
SP3-02（デモプラン）。**ADR 承認**（OpenCode 統合方式 — adr_approval ハードゲート）。
大阪 OpenAI 互換 agentic API（CLAUDE.md 記載のベースURL/モデル）を OpenCode のバックエンドに使う想定。

## 作業内容（ADR 承認後・specs/19 で確定)
- ADR 冒頭検証: OpenCode headless の実挙動・LLM 認証経路（署名プロキシ等）・ランタイムの成立を
  実機確認して ADR を起草（specs/19 §4.4）→ 人間承認。
- 生成オーケストレーション（specs/19 §4.5 — `POST /api/builder/sessions/{sid}/generate`・
  202 + 非同期・Demo を `provisioning` で内部作成・`failed→provisioning` の再実行契約・
  フェーズ境界の status 再確認・§8.2 のリース区間）。
- **安全ゲート（specs/19 §4.3、fail-closed）**: 隔離サンドボックス（S1）・資格情報ゼロ（S2）・
  スコープ担保の多層（S3 — 固定 API クライアント/バンドル静的検査/配信時 CSP）・
  秘密混入検査（S4）・staging → ポインタ切替の原子的公開（S5）・台帳 write-ahead（S6）。
- 保管・配信（specs/19 §5 — `demo-bundles/<sha1(namespace)>/<bundle_id>/` 保管・versioning
  Disabled 必須・`GET /api/demos/{id}/app/{path}` 配信 = require_demo + ready ゲート・
  bundle_id の UUID 形式検証とトラバーサル 404・CSP/nosniff・Content-Type 許可リスト）。
- `Demo.config` のサーバ管理キー（specs/19 §5.3 — plan/frontend/generation は PATCH 422・温存マージ）。
- DELETE 後始末への **3g 追加**（specs/19 §5.4 — バンドル prefix のページネーション完走削除・
  NotFound 以外 503・台帳スキップ判定）。
- テスト: fixture バンドルでの静的検査（スコープ外参照・生 fetch・秘密パターン → 不合格）・
  配信のパス検証/ヘッダ・config 温存マージ/422・後始末 3g・生成骨格（LLM/OpenCode はモック）。
