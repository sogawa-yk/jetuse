# タスク: SP3-03 フロント生成（OpenCode + OCI モデル→静的SPAバンドル）+ デモ配信

> 受け入れ条件は specs/19 準拠で確定（SP3-00 起草。**specs/19 の人間承認が着手条件**）。以下は暫定。
> **本タスクは技術リスクの核**。実装前に **ADR（OpenCode 統合方式）を起草→人間承認**（adr_approval ゲート）。

## 目的
デモプラン（SP3-02）から、OpenCode + OCI モデルで**静的SPA バンドル**を生成し、デモごとに保管して
`/api/demos/{id}/...` で配信する。生成フロントは JetUse API をデモスコープで叩くだけ（バックエンド生成なし）。

## 仕様参照
specs/19-sp3-builder.md（フロント生成の要件・非機能・安全要件）、specs/17 §1・§5・§6。
**ADR（本タスクで起草）**: OpenCode 統合方式（ランタイム・サンドボックス・生成物の安全担保）。

## 対象 area
api（生成オーケストレーション）。生成物は静的SPA（web ビルド成果物に相当するが人手では書かない）。

## 前提（依存タスク / 人間の事前作業）
SP3-02（デモプラン）。**ADR 承認**（OpenCode 統合方式 — adr_approval ハードゲート）。
大阪 OpenAI 互換 agentic API（CLAUDE.md 記載のベースURL/モデル）を OpenCode のバックエンドに使う想定。

## 作業内容（ADR 承認後・specs/19 で確定・暫定）
- ADR で決めた方式で OpenCode をサーバ側から起動し、デモプランを入力に静的SPA を生成する。
- **安全担保（構造的に）**: 生成コードは JetUse API の**そのデモの `/api/demos/{id}/...` スコープしか
  叩けない**・秘密/資格情報を埋め込ませない・生成物はサンドボックスでビルド。生成失敗/危険検知は fail-closed。
- 生成バンドルをデモごとに保管（object storage 想定）し、`/api/demos/{id}/...`（PAR/署名配信）で開ける。
- `Demo.config` / status を生成状態に合わせて更新（provisioning→ready/failed、SP2 の状態機械）。
- テスト: プラン→生成の骨格（LLM/OpenCode はモック可）・生成物が範囲外エンドポイントを含まない検査・
  秘密混入の静的検査・配信パスの越境404。
