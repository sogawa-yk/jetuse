# タスク: SP3-04 サンプルデータ生成 + 箱への投入（demo_<id>）

> 受け入れ条件は **specs/19 §6（+ §9 SP3-04）で確定**(SP3-00 起草済み。
> **specs/19 の人間承認が着手条件**)。

## 目的
デモプラン（SP3-02）のデータ定義から、サンプルデータを生成してデモの箱 `demo_<id>`
（SP2 の論理名前空間 — specs/18 §3.1）へ表作成＋投入し、文書を RAG 箱へ投入する。
dbchat / rag.search がそのデータを引ける状態にする。

## 仕様参照
specs/19-sp3-builder.md §6（データ生成/投入・冪等置換・隔離）、specs/18 §3（箱のプロビジョニングと
後始末・排他リース・上限）・§4.3（デモスコープ dbchat）、specs/17 §5・§6。

## 対象 area
api

## 前提（依存タスク / 人間の事前作業）
SP3-02（デモプランのデータ定義）。SP2-02 の箱プロビジョニング/後始末が土台。

## 作業内容（specs/19 で確定）
- 表データ投入（specs/19 §6.1 — **`create_dataset(namespace, …)` 系の既存内部関数のみ**で投入。
  registry-first・VPD・完全ハッシュ命名・リース・上限の既存契約を素通しする。新規の投入経路・
  命名・後始末部品を作らない。既存 `generate_dataset` 系の流用を第一候補に）。
- LLM 行データ生成（specs/19 §6.1 — プラン準拠の型・件数をサーバ側で検証、有界再試行 →
  不合格なら生成失敗 = `failed`）。
- 文書投入（specs/19 §6.2 — LLM が Markdown 生成 → **デモ RAG upload の既存内部関数**で投入。
  1 文書 ≤ 64KB）。
- 冪等置換（specs/19 §6.3 — 同名 dataset / 同名文書は「外部先行の既存削除 → 再作成」。
  failed → 再生成で収束）。
- 単体テスト（fake LLM / fake 外部）: 型不一致の再試行 → 失敗・同名置換・
  owner キーが `ctx.namespace` 固定（他デモ/共有スキーマ/user 資産に触らない）。

## 完了条件（検証可能な述語で — specs/19 §9 SP3-04 と同一）
- [ ] デモプランから `demo_<id>` に表＋サンプルデータ、RAG 箱に文書が投入され、
  デモスコープ dbchat / rag.search がそのデータを引ける。
- [ ] 再生成が冪等（同名置換で収束）。デモ削除で本タスク由来の表・文書・store が後始末される
  （孤児なし — specs/18 §3.2 + specs/19 §5.4 の既存後始末で。専用部品なし）。
- [ ] 単体テスト全緑・ruff クリーン・回帰なし。
- [ ] 実環境 E2E（プレビュー環境・実 ADB・`demo_<id>` 隔離）: プラン → 投入 → dbchat 日本語照会 →
  rag.search 照会 → デモ削除で消えるまでを `runs/<run-id>/e2e/` に記録。

## E2E シナリオ
- プレビュー環境で、プラン1件 → `demo_<id>` 表作成/投入 + 文書投入 → dbchat で自然言語照会・
  rag.search で文書照会 → デモ削除で後始末完走（表・文書・store・prefix ゼロ）。
