# タスク: SP3-05 ビルダー UI（ヒアリング→プレビュー→保存）+ デモ産出 E2E

> 受け入れ条件は **specs/19 §7（+ §9 SP3-05・§10）で確定**(SP3-00 起草済み。
> **specs/19 の人間承認が着手条件**)。

## 目的
フィールドSA 向けのビルダー UI を作り、ヒアリング→デモ設計→生成→**プレビュー→確定**の一連を
1画面のウィザード体験にまとめる。パイプライン（SP3-01〜04）を束ねて `Demo` を産出する縦切りを完成させる。

## 仕様参照
specs/19-sp3-builder.md §7（ビルダー UX・産出フロー）・§10（ステージ完了条件）、
specs/17 §1・§6、旧 UC-03（非開発者が5分で作成）。

## 対象 area
web（+ 束ねる薄い api glue があれば api）

## 前提（依存タスク / 人間の事前作業）
SP3-03（フロント生成・配信）＋ SP3-04（データ投入）。

## 作業内容（specs/19 で確定）
- ビルダー UI（specs/19 §7 のウィザード）: ①ヒアリング（NL 対話・必須項目の充足可視化）→
  ②プラン確認（要約表示。**JSON 直接編集なし** — 修正は追加発話 → 再 design。タイトル/説明のみ
  直接編集）→ ③生成進行（202 + ポーリング。failed は理由 + 再生成ボタン）→
  ④プレビュー（`/api/demos/{id}/app/` を開く）→ ⑤確定（PATCH name/description）/ 破棄（DELETE・確認付き）。
- 直近セッションの復帰は localStorage（一覧 API は residual — specs/19 §2.4）。
- Internal 面（AUTH 前提）。既存 SPA のデザイントークン・コンポーネント流儀（UI-02 等）に準拠。
- UI ロジックの単体テスト（状態遷移・エラー表示・failed → 再生成導線）。

## 完了条件（検証可能な述語で — specs/19 §9 SP3-05 と同一）
- [ ] ビルダー UI からヒアリング → 設計 → 生成 → プレビュー → 確定が通り、確定後に一覧
  （SP2 CRUD）へ現れる。
- [ ] `npm run build` 緑・UI 単体テスト緑・既存 web 回帰なし。
- [ ] **デモ産出 E2E（プレビュー環境・ステージの総合シナリオ — specs/19 §10）**: 「フィールドSA が
  ヒアリングに答える → デモ設計 → 生成 → データ投入 → ready → 生成デモが `/app/` で開き
  chat/rag/dbchat をデモスコープで叩ける」をブラウザで実機確認し `runs/<run-id>/e2e/` に記録
  （旧 UC-03「5分で作成」を Internal ビルダーで充足）。

## E2E シナリオ
- プレビュー環境（gateway）に対し、ビルダー UI から新規デモを1件作り切り、生成デモの動作
  （デモスコープ chat/rag/dbchat）まで確認。

## 成果物
- コード（ビルダー UI・glue・テスト）・`docs/verification/SP3-05.md`（ステージ総合 E2E 証跡込み）。

## 非ゴール / 禁止事項
- マーケット公開/配布（SP4）。バックエンド生成。プラン JSON の自由編集 UI（specs/19 §11）。
- 認証情報のコミット禁止。
