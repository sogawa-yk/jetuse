# タスク: SP1-01 能力ディスクリプタ8件 + GET /api/capabilities

## 目的
ビルダー（SP3）が読む機械可読な「能力カタログ」を提供する。自動生成 OpenAPI（技術契約）に、デモ向け
能力へ手書きディスクリプタ（用途・例・安全フラグ）を重ねる（specs/17 §3「案1」）。
**能力追加のコスト = ルート追加 + ディスクリプタ1件** をここで確立する。

## 仕様参照
specs/17-demo-platform-redesign.md §3（能力カタログ）・§4（公開する能力）

## 対象 area
api

## 前提（依存タスク / 人間の事前作業）
なし（第1波・SP1-02 と並列可）。

## 作業内容
- `packages/api/jetuse_core/capabilities.py` を新設し、8 能力のディスクリプタを**データ**（dict のリスト）
  として定義する。フィールド（specs/17 §3）:
  - `capability`（`chat` / `rag.search` / `dbchat` / `agents` / `voice` / `minutes` / `translate` / `docunderstand`）
  - `summary`（何ができる・1行）
  - `when_to_use`（デモでの使いどころ。SP3 の LLM が能力選択に使う）
  - `example`（代表的な入力例 → 出力の要点。dict）
  - `demo_safe`（bool。本タスクでは 8 件すべて true）
  - `routes`（対応ルート。`[{"path": "/api/chat", "method": "post"}, ...]` の形。
    translate/docunderstand の実ルートは `routes/voice.py` 内にある — path を実装時に確認して記載）
- `packages/api/service/routes/capabilities.py` を新設: `GET /api/capabilities`（`require_user`）。
  各ディスクリプタに、`app.openapi()` から対応 path/method の requestBody / responses スキーマ断片を
  結合して返す。レスポンス形: `{"capabilities": [{...descriptor, "openapi": {...}}]}`。
  **この出力形はカタログの安定契約**（将来「案2」へ内部を差し替えても形は変えない — specs/17 §3）。
- `service/main.py` に router を include。
- 単体テスト `packages/api/tests/test_capabilities.py`:
  - 8 件が返り、`capability` 名が仕様 §4 の集合と一致する。
  - 各ディスクリプタの `routes` の path/method が `app.openapi()["paths"]` に**実在**する
    （ディスクリプタとルートの乖離をテストで防ぐ — カタログの正しさの要）。
  - 裏方（`/api/admin` / `/api/conversations` / `/api/tools` 等）がカタログに含まれない。
  - 認証必須（`AUTH_REQUIRED=true` 相当のモックで 401）。

## 非ゴール / 禁止事項
- 統一 Capability インターフェース（案2）は作らない。ディスクリプタは素の dict でよい
