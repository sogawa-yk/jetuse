## 実装要約

- **auth.py fail-closed 強化（specs/18 §5.1 — codex review-2 B006）**: `AUTH_REQUIRED=true` のとき
  `OIDC_ISSUER` / `OIDC_AUDIENCE` / `OIDC_JWKS_URL` の**3点全てを必須**とし、いずれか未設定なら 500
  （検証を欠いたまま受理しない。同一 IdP の別アプリ用トークンの受理を防ぐ）。従来の
  「issuer/audience が空ならその検証を無効化」「sub 欠落でも subject='' で受理」を廃止し、
  検証後の `sub` 欠落/空は 401。空白のみの設定値も未設定扱い（strip — review-3 m001）。
  共有関数 `verify_token` 1箇所の修正で FastAPI（require_user）と Fn ルーター（ARCH-02）の
  両経路に効く。
- **テスト（TDD RED→GREEN）**: `tests/test_auth.py` 新設 — RS256 実署名（ローカル鍵・JWKS 取得のみ差替）で
  wrong issuer 401 / wrong audience 401 / missing・empty sub 401 / 3点いずれか欠落 500（空文字・
  空白のみの両方）/ 正常系 subject 抽出。非文字列 sub（123/true/配列/オブジェクト）も 401
  （isinstance ガード — codex review-1 m001）。
- **.env.example**: 固定 issuer 行をプレースホルダ化（`<discoveryのissuerをそのまま>`）・`OIDC_AUDIENCE` 行追記
  （audience は discovery から得られない旨のコメント付き — specs/18 §5.1）。
- **Internal 配備の AUTH_REQUIRED=true 既定化**: `infra/terraform/environments/dev/main.tf` の locals で
  `api_environment = merge({ AUTH_REQUIRED = "true" }, var.api_environment)`（CI・Fn ルーター両方に適用。
  tfvars 側で明示上書き可）。tfvars.example に OIDC 実値の投入例を追記。さらに
  **AUTH_REQUIRED=true かつ OIDC 3値不備の apply を plan 時に停止**する precondition ガードを追加
  （「3値を投入し忘れて配備後に全滅」を plan で検出 — review-3 M002。判定は Pydantic の bool 真値と
  同義の truthy 正規化（"TRUE"/"1"/"yes" 等でも発火 — review-4 M002）。iac-validate.log [dev-1〜4]）。
  **コード上の `Settings.auth_required=False` は不変更**（Public/main 既定を壊さない）。
- **ORM スタックの OIDC 契約（codex review-1 B001 / review-2 B001・M001）**: infra/orm は
  `enable_auth=true` 既定のまま `OIDC_AUDIENCE` を注入せず、issuer/JWKS も固定値だった。
  `oidc_audience` / `oidc_issuer` / `oidc_jwks_url` 変数（schema.yaml 入力欄付き）を追加して注入。
  issuer は**固定既定なし＝discovery の issuer を必須入力**（review-4 B001。固定値をコミットしない —
  specs/18 §5.1）。JWKS のみ自作成ドメインの SigningCert パスへフォールバック（空白のみ入力も
  trimspace でフォールバック — review-4 M001）。**enable_auth=true かつ audience/issuer 空は plan 時に停止**
  （`terraform_data.oidc_config_guard` の precondition。変数間 validation は TF≥1.9 のため
  required_version 1.5 互換の precondition を採用 — review-2 M001。ランタイムの無言全滅より plan 時
  fail-closed。audience は discovery から導出できず実測でしか得られないため、値の捏造はしない）。
  検証: `terraform validate` 通過 + plan 時ゲート3系統（空で停止 / 提供で通過 / enable_auth=false で通過）=
  `runs/2026-07-07T0609_SP2-04/e2e/iac-validate.log`。SPA が resource scope 付きトークンを取得する
  ORM 側の完全な OIDC resource 登録と認証有効 ORM 配備の実トークン E2E は Public 側の後続タスク

## residual（非 blocker — 後続 / 人間トリアージ）

- **ORM ワンクリックの認証有効経路の完全化（codex review-3 B001/M001）**: ORM が自作成する SPA アプリは
  OAuth resource（primary audience / resource scope）を登録せず、SPA も `openid profile email` しか
  要求しないため、認証有効の ORM 配備で SPA トークンが新 audience 契約を満たす保証がない。
  決定的な resource scope 登録 + SPA の scope 要求 + discovery 由来 issuer/JWKS の自動導出は
  **SPA 認証フロー（Public 面）の変更**であり、SP2-04 の非ゴール「Public（main）の認証既定の変更」に
  抵触するうえ、検証には認証有効 ORM スタックの実 apply（人間ゲート）が必要。検証不能なまま推測で
  SPA フローを改修すると「直ったように見えて壊れている」を作るため、本タスクでは
  **上書き変数 + plan 時 fail-closed ゲート**（不備配備の阻止）までとし、完全化は Public 側の
  後続タスクとして起票する（infra/orm は main 起点 PR が必要な Public 面 — SP2-03 の層2ゲート
  main 展開 residual と同型）。
- SP2 dev 枝の auth.py 強化の main（Public）バックポート（層2ゲートと同じ main 起点 PR に同乗可）。
