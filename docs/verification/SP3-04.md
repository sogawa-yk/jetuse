## 実装の要点(§6 との対応)

| 仕様 | 実装 |
|---|---|
| §6.1 表 = 既存 datasets 機構のみで投入 | `datasets.create_dataset` を直接使用(registry-first・VPD・完全ハッシュ命名・リース・箱上限を素通し)。プラン列型は `column_types`(既存関数の後方互換 opt-in 引数。許可リスト検証 + DATE/TIMESTAMP/NUMBER の値変換)で物理スキーマへ反映 — 新規の投入経路・命名・後始末部品なし |
| §6.1 LLM 行データ生成 + サーバ側検証 | `generate_dataset` 系と同じ GEN_MODEL(gemini-2.5-flash)。ヘッダ完全一致・件数一致・型検証(NUMBER=Decimal 有限値 + NUMBER(p,s) 整数部桁 / DATE=ISO / TIMESTAMP=ISO / VARCHAR2(n CHAR)=n 文字)を有界再試行(3 回)→ 不合格は `DataProvisionError`(usage 込み)= 呼び出し側で `failed` |
| §6.2 文書 | LLM Markdown → `rag.add_file`(予約 ledger・quota・store lazy 生成の既存経路)。1 文書 ≤ 64KB(超過・空は再試行)。投入後、vector store 索引の completed を有界待機(300 秒)— rag.search が引ける状態が受け入れ条件のため fail-closed |
| §6.3 冪等置換 | 同名 dataset = `delete_dataset`(外部先行 DROP)→ 再作成 / 同名文書 = `delete_file`(外部先行)→ 再 upload。**生成が検証を通った後にのみ削除**(生成失敗で既存データを消さない) |
| §6.4 隔離 | owner キーは `demo_<id>`(DemoContext.namespace と同一導出)固定。単体テストで全呼び出しの owner を検証 |
| §8.2 リース | LLM 生成はリースを跨がない: フェーズ1(生成・検証=リース外)→ フェーズ2(置換+投入= `demo_lease.mutation` 保持・status 再確認込み)→ フェーズ3(索引待機=読み取りのみ・リース外) |
| §8.3 usage | 全試行の usage を合算して戻り値 / `DataProvisionError.usage` に載せる(記録は呼び出し側= SP3-03 が実ユーザーに紐づけて usage_log) |

### 実機所見(docs/tips.md 追記済み)

**大阪リージョンの vector store 障害(2026-07-08・サービス側)**: store が作成後 12〜18 秒で
expired になり completed に達しない(空/file_ids 付きとも)。同一コードで Chicago は 48 秒で
completed。プローブ 5 本の切り分けは `e2e/osaka-vector-store-outage.md`。このため大阪での
rag.search 照会は SKIPPED(`e2e/SKIPPED.md`)とし、文書経路は (a) 大阪で fail-closed 実証 +
(b) Chicago でフル実機、の 2 面で検証した。
