## 1. 課題と実装

OKE 素デプロイで「サンプル NL2SQL がデモ起動だけでは動かない（データ表が実体化されず ORA-00942）」
という本質ギャップ（memory ②）を塞ぐ。従来 `synth._seed_plan` は plan-only（行数を数えるだけ）、
実テーブル生成＋seed＋GRANT は ENH-01 の CSV 経路（`datasets.py`）にしか無く、デモ起動から呼ばれて
いなかった。

実装（最小差分）:
- **新規 `jetuse_core/materialize.py`**: `SampleAppDefinition` の datasets を
  「**接続ユーザ自身のスキーマ（adb_user）** へ `CREATE TABLE`（列型は fields の宣言型→Oracle 型。
  string は **常に `VARCHAR2(4000 BYTE)` 固定**＝replace_later 顧客データ差替えで ORA-12899 を出さない /
  seed 変更で列幅が動かない）→ seed 投入（date/datetime/number/
  boolean を coerce）→ 読取ユーザ（adb_query_user）へ `GRANT SELECT`」する。別スキーマへ DDL すると
  最小権限環境で ORA-01031 になるため作成先は接続スキーマに固定。テーブル名は dataset 名（大文字）で
  NL2SQL の許可テーブル `ds.name.upper()` と 1:1。
  - **作成先 = 読取先 = `target_schema()`**（単一の出所。= `SAMPLE_DB_SCHEMA` か未設定なら adb_user）。
    作成は接続ユーザ自身のスキーマでしかできないため、`target_schema()≠adb_user` は
    `MaterializeConfigError` で**起動時に明示エラー**（作成先と読取先の不一致による ORA-00942 を未然に
    検出。別スキーマ展開は未対応）。
  - **起動は非破壊**: 既存の管理下 ready 表を **DROP しない**（seed 方針が別でも既存データを壊さない）。
    再構築は明示 `recreate=True` か、**不完全(pending) / 形 fingerprint 不一致**のときだけ。`seeded` は
    fingerprint に含めない（replace_later 起動が既存 seeded データを消さない）。
  - **seed 状態の追跡**: レジストリに seeded を記録。空のまま作った表（seeded=False）へ後から
    seeded=True が来たら、**空のときに限り** seed を注入（False→True / True→False 両順序で非破壊）。
  - **所有権レジストリ `JETUSE_MATERIALIZED_DATASETS`**: 自分が作った表だけを reuse/recreate/GRANT
    対象にし、管理外の同名物理表（他者の表）や **別 owner（別アプリ）** の同名表は触らず
    `MaterializeConflictError`（fail-closed。別アプリのデータ混在/誤再利用/無断 DROP を防ぐ）。各表に
    **owner / status(pending|ready) / fingerprint**（fields＋固定列型＋仕様版）を持ち、途中失敗した
    不完全表は次回安全に再構築（回復性）。
  - **物理表に焼く所有マーカー（table comment `JETUSE_MAT:<owner>`）で全経路を二重防御**（F-002）:
    レジストリ（別表）の claim **だけ**を信用しない。**物理表が存在する全経路（reuse / GRANT / seed /
    DROP）で、触れる前に物理表自身のマーカーで `owner` を検証**する。マーカー無し／別 owner の表は
    `MaterializeConflictError`（外部置換された ready 表を読取ユーザへ GRANT で晒さない／他者データを
    DROP しない）。新規作成中に同名表が外部出現（ORA-00955）したら claim を取り消し fail-closed。
    recreate=True でも物理マーカー不一致なら DROP しない。
  - **非空 × 形（fingerprint）不一致は明示 conflict**（起動失敗）: schema ドリフトしたデータ付き表を
    silent reuse で古い形のまま ready に戻さない。`recreate=True` か migration を要求する。pending（作りかけ）
    でデータがあり形が一致する場合とは分岐し、後者は ready に確定して reuse（回復）。
  - **owner 正規化**（F-004）: owner 空は Oracle で空文字→NULL となり所有比較が壊れるため、空は安定 sentinel
    `_unspecified` に正規化し、owner_id 列（VARCHAR2(200)）超の owner は予測可能な ValueError。
  - **後方互換の launch**（F-003）: `SAMPLE_DB_SCHEMA≠adb_user`（事前プロビジョニング/legacy 運用）は
    `materialize_app`（launch 経路）が auto をスキップして起動を壊さない（読取は従来 `SAMPLE_DB_SCHEMA`）。
    直接 `materialize_definition` 呼び出しの config error はプログラム誤用の fail-closed として温存。
  - **直列化**: 同一 schema.table を `DBMS_LOCK` で直列化（同時起動の部分作成・二重 DROP を防止）。
    **DBMS_LOCK 不可は fail-closed**（bootstrap.py / ops/setup-dev-schema.py が GRANT EXECUTE 済み）。
  - **対象外**: 専用外部スキーマで運用するアプリ（SBA-C / `nl2sql_schema=JETUSE_SBA04`≠target_schema）は
    `materialize_app` がスキップ（誤った adb_user 表を作らない。SBA-C の provisioning は SBA-04 の責務）。
  - **列型**: string は **常に `VARCHAR2(4000 BYTE)`**（seed 値で幅を狭めない＝差替えで ORA-12899 を出さない /
    列幅が fingerprint に影響しない）、`BYTE` セマンティクスで NLS 非依存。4000 BYTE 超の string seed は
    予測可能な ValueError（長文は type=text→CLOB に倒す）。fingerprint は **seed 非依存**（fields の
    名前/固定列型＋仕様版のみ）で、サンプル変更だけで既存表の再構築を誘発しない。
  - **seed 方針**: 合成の `SeedPlan.seeded` を反映（`seeded=False`= replace_later/genai は表だけ作って
    seed しない）。
- **`service/routes/hearing.py`（`/launch`）**: デモ起動時に `materialize_app(instance_id,
  seeded=composition.seed.seeded)` を実行。失敗は握りつぶさず伝播（一過性 DB→503 / それ以外→500。
  200 で成功偽装しない）。`record_launch` の前に実行し、失敗時は起動記録を残さない（成否境界の原子性）。
- **`service/routes/sample_apps.py`（`/dbchat/execute`）**: 読取の `CURRENT_SCHEMA` 固定先を
  `materialize.target_schema()` に一本化（作成先と読取解決先を整合。従来の未設定時 None からの意図的
  変更・回帰テスト追加）。

CSV アップロード経路（`datasets.py`）は非ゴールにつき改変せず再利用に留めた。
