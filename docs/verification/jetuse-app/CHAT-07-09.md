## CHAT-07: DB接続タイムアウト + 503即時返却（backlog #11）

朝の実障害（夜間停止でADBがSTOPPED→送信UI完全無反応）の再発防止。

- db.py: `tcp_connect_timeout=5s` / `POOL_GETMODE_TIMEDWAIT, wait_timeout=5s` / `ping_interval=30s` / 接続に `call_timeout`（既定10s、`DB_CALL_TIMEOUT_MS`）。ウォレット取得失敗も `oracledb.OperationalError` に正規化
- main.py: `oracledb.Error` 例外ハンドラ → **503 `database unavailable`**
- フロント: 会話作成15sタイムアウト、失敗時はステートレス継続+「履歴は保存されません」通知。SSE 503はバブルにメッセージ表示

**実機（ADBを実際に停止して検証）**:
- ADB STOPPED中: `POST/GET /api/conversations` → **503を約5秒で返却**（従来は無限ハング）
- ADB STOPPED中: ステートレスチャット（conversation_idなし）は**正常動作** = ユーザーはDB障害中も会話を継続できる
- ADB再起動後: **CI再起動なしで自動復旧**（会話作成200。プールのping/再取得が機能）

## CHAT-08: クライアント切断で上流LLMストリームを打ち切る（backlog #2）

- SSEジェネレータのfinallyでキャンセルフラグ→produceスレッドが検知し上流`close()`。部分応答はADB保存維持。満杯キュー時のデッドロック対策（put 1sポーリング+キャンセル検知）込み
- chat.py: `_stream_responses` / `_stream_chat_completions` にfinallyで`stream.close()`
- pytest: 実uvicornを立てて実切断を再現（TestClientは切断を模せない）。切断後10s以内に上流closeを確認
- **実機**: 「1〜500を書き出す」ストリームを4秒で切断 → CIログに `"upstream cancelled", partial_chars=38` が出力され、以後のchat_doneなし = **完走せず打ち切り**（API GWがクライアント切断をバックエンドに伝搬することも確認）

## CHAT-09: 会話削除時のOCI Conversation削除同期（backlog #3）

- DELETE時に`oci_conversation_id`があればADB削除成功後にOCI側をベストエフォート削除（失敗はログのみ）
- **実機**: 短期メモリ付き会話を作成→OCI側retrieveで存在確認→API DELETE→**OCI側404**を確認
