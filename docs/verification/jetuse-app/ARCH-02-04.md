## ARCH-02: 非ストリーミングAPIのFunctions移行（第1陣）

| 項目 | 内容 |
|---|---|
| 実装 | `fn/router/func.py`（単一ルーター方式、jetuse_core共用、JWT検証は`auth.verify_token`へ共通化）。担当: presets / dbchat(schema・execute・chart) / tts の7本 |

| インフラ | functionsモジュールに`oci_functions_function`(512MB/120s)追加、GWの`functions_routes`へ3セグメント配線。CIは温存（移行期間のフォールバック） |

### 実機で確定したハマりどころ（tips記録済み）

1. **GWルートの`{p*}`は残りセグメント無しのパス（`/api/presets`）に一致しない** → 完全一致ルート併設
2. **GW→Functionsは専用IAMポリシー必須**（無いと一律500。function直接invokeは正常なので紛らわしい）
3. ADB夜間停止の再発と重なり切り分けが複雑化 → backlog #10は「手動運用（エージェントは作業時に起動）」で判断確定

## 実測値（GW経由・E2E、2026-06-13）

| 測定 | 結果 |
|---|---|
| **コールドスタート**（コンテナ起動+JWT+ADBウォレット取得+mTLS接続+クエリ） | **2.07秒** |

| ウォーム dbchat/execute（実SQL） | 0.17s |
| CI側SSE（チャット）への影響 | なし（ルート優先度どおり分離動作） |

→ 懸念だったコールドスタートは**2秒強**で、CRUD系の初回として実用域。
provisioned concurrencyは不要と判断（コスト増に見合わない）。

## ARCH-03: 常駐系の最適化

- CI右サイズ: **1 OCPU / 8GB → 4GB**（適用済み。SSE+STT+大容量アップロード用途には十分）
- ADB夜間停止: **手動運用に決定**（2026-06-13ユーザー判断。セッション開始時のADB確認を標準手順化 — backlog #10）
- OKE等への載せ替えは不採用（ARCH-01の比較どおり）

## ARCH-04: 移行前後の定量比較

| | 移行前 | 移行後 |
|---|---|---|
| 構成 | CI 1OCPU/8GB に45本同居 | CI 1OCPU/4GB（SSE/状態/大容量=38本）+ **Functions（7本、512MB）** |
| 月額（compute） | $27.4 | **$23.0**（CI $23.0 + Fn無料枠内$0 + GW無料枠内$0） |
| ウォームレイテンシ | 数十ms | 同等（35〜60ms） |
| 初回レイテンシ | なし | コールド2.1秒（アイドル後初回のみ） |
| 障害分離 | 全滅リスク（2026-06-12に実例） | チャット系障害時もpresets/dbchat/ttsは生存（逆も同様） |
| デプロイ | 全API再作成（壊してから作る） | fn側は関数単位の差し替え（CI無停止） |
