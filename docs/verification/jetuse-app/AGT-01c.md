## 追加ツール（アプリ既存機能の自己統合）

| ツール | 実体 | 承認 |
|---|---|---|
| **rag_search**（社内文書検索） | file_search built-in + ユーザーのVector Store（選択時のみtools注入） | 不要（built-in） |
| **query_database**（DB照会） | SQL Search生成→JETUSE_QUERY読取専用実行（SQL-02の多層ガード再利用） | **不要**（読取専用） |
| **get_current_time** | JST現在日時 | 不要 |
| web_search / web_fetch | 既存 | 必要（外部送信） |

- **承認パーティション**: requires_approval=Falseのツールは都度承認モードでもサーバー側自動実行（混在バッチは全件承認にフォールバック — ステートレス継続での結果消失防止）
- **MCPプリセット**: deepwiki / **Microsoft Learn**（匿名動作を実機確認）をワンクリック登録チップとしてUI追加

## 安定性対策（同時実施）

- keepaliveをコメント行→**dataフレーム**に変更（経路上の特別扱い疑いの排除実験。切断は5連続中1件に減少も根絶せず → GW/経路側の事象と結論、backlog #12継続）
- クライアントリトライ条件を「本文テキスト未受信」基準に緩和（ツール通知のみの段階なら読み取り専用ツールのため安全に再実行。途中通知はリセット）

## 既知の軽微事項

- rag_search実行中の「実行中」通知イベントが表示されないことがある（file_search_callのストリームイベント形式が想定と異なる模様。機能・回答には影響なし）

## 追補（AGT-01d、ユーザー報告2件の修正 — イメージ0.18.0/migration 008）

1. **エージェント定義に「ツール自動実行」がない** → `agents.auto_tools` 列を追加（migration 008）。ビルダーにチェックボックス、チャット時はエージェント定義の値を使用
2. **リサーチエージェントが10回ループして回答しない** → 根本原因は**承認ラウンドをまたぐとUIが最新ラウンドのツール結果しか送らず、モデルが過去の検索結果を毎回忘れて再検索していた**バグ。修正: UIがターン内の全ツール結果を累積送信（turnResultsRef）。加えてサーバー側に二重の安全弁: ①ホップ上限到達時はエラーではなく**ツールを外して「手持ち情報で最終回答せよ」を強制** ②累積ツール結果16件で同様に打ち切り
- 再現E2E: ユーザー報告と同一のinstructionsを持つリサーチエージェント（自動実行）が、ツール5回使用後に強制回答へ切替わり、**実データ入りの構造化リサーチ結果を返すことを確認**
