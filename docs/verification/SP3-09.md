## 1. 何を作ったか

ORASEJAPAN 共有テナンシ（gpt-5 系）の API 署名鍵材料を、RM スタックの sensitive 変数
（SP3-07 の `GEN_SHARED_KEY_PEM_B64` 等）から **jetuse:dev の OCI Vault シークレット**へ移した。
デプロイされた API はリソースプリンシパルで取得し、鍵をディスク・環境変数に出さず **in-memory 署名**する。

- **`infra/.../dev-app/vault.tf`**（新規）: Vault（DEFAULT）+ **software 保護キー**（追加課金なし）+
  シークレット（JSON: `user`/`tenancy`/`fingerprint`/`region`/`key_pem`）。実値は tf に置かず
  placeholder `"{}"` で作成し `ignore_changes=[secret_content]`。3 リソースに
  `prevent_destroy=true`（destroy 禁止を計画で強制 — review-2 M002）。
- **`jetuse_core/gen_shared_vault.py`**（新規）: RP secrets クライアントで CURRENT 版を取得 →
  必須キー検証 → `oci.signer.Signer(private_key_content=...)` を in-memory 構成。成功はプロセス
  生涯キャッシュ、失敗は 30s 負キャッシュ（フェッチ嵐防止）+ `threading.Lock` で single-flight。
  取得は SDK 同期 I/O のため **`sign_proxy` が `asyncio.to_thread` で off-loop 実行**（イベント
  ループ非閉塞 — review-1/2 M001）。user principal は期限切れしないため周期/401 の自動再取得は
  封じ（`refresh_interval` 無限大 + `_refresh_signer` no-op）、反映は API 再起動。
- **`sign_proxy._route`**: 共有モデルの auth 優先順位 = `GEN_SHARED_SECRET_OCID`（Vault・デプロイ）>
  `GEN_SHARED_PROFILE`（~/.oci・ローカル）。Vault 取得失敗はプロファイルへ **fallback しない**
  fail-closed（403 model_not_configured）。設定は非鍵材料 `GEN_SHARED_SECRET_OCID` 1 個へ集約。
- **`entrypoint.sh`**: SP3-07 の `~/.oci` プロファイル書き出しブロックを撤去（鍵材料を env で
  受け取らない）。**`ops/deploy-dev-app.sh seed-env`**: ~/.oci の ORASEJAPAN プロファイルを読んで
  Vault へ新版投入（`--wait-for-state ACTIVE` — review-2 OPS-001。実値は argv/RM 変数に出さない）+
  旧 RM 鍵材料 6 変数を除去 + 冪等性は**実内容比較**で判定（review-2 m001）。

## 2. 実機で発見した事実（Tips — docs/tips.md にも追記）

1. **oci SDK の SecretsClient は RP signer でも region 必須**（"Must supply either a region or an
   endpoint"）。`{"region": oci_region}` を明示（db.py の流儀）。クライアントのモックでは捕まらない
   ので構築を 1 回は実 SDK で通すこと（初回イメージがこの見落としで Vault 経路 fail-closed のところ、
   修正版で解消）。
2. **`container-instance restart` は非同期** — 発行直後の `/api/health` は旧コンテナの 200 を拾う。
   `--wait-for-state SUCCEEDED` を待たないと実再起動が生成 BackgroundTask を殺す（ローテーション
   E2E take1 で実害）。
3. RM スタック変数は構成 zip から宣言を消しても残る → `stack update --variables` で明示 `del`。
   `oci vault secret update --secret-content file://...` なら鍵材料が argv に出ない。
4. RP による secret-family 読取は既存 DG jetuse-internal-dg（manage all-resources）で成立 =
   **IAM 変更不要**（実機ログで確認）。
