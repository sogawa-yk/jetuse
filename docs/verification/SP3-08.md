## 1. 何を作ったか

フロント生成の実行環境を host podman 近似から **ADR-0023 §1 の本来設計 = 生成ごとの使い捨て
OCI Container Instance（2 相）** へ移行し、**jetuse:dev にデプロイされた API 上でビルダーの
「生成」が end-to-end で動く**ようにした（descope 項目①の解消）。

- **`jetuse_core/generation_runtime_ci.py`**（新規）: 相1 生成 CI
  `jetuse-builder-gen-<job_id>`（OpenCode + vendored scaffold・非信頼）→ API 側で敵性 tar 検証
  （symlink/hardlink/絶対/`..`/重複/数/サイズ fail-closed・保護原本除去）→ 相2 信頼ビルド CI
  `jetuse-builder-build-<job_id>`（OpenCode 非搭載・node_modules 焼き込み）→ dist 受領。
  shape = CI.Standard.E4.Flex 1ocpu/4GB・restart NEVER・`is_resource_principal_disabled=true`。
- **鍵レス受け渡し**: rag_bucket の `jetuse-builder-jobs/<job_id>/` prefix + **期限付き
  オブジェクト単位 PAR**（読取専用/書込専用を相・方向で分離）。CI に OCI 資格情報・RP・
  pull secret を一切渡さない（イメージは public OCIR repo の匿名 pull — ADR-0011）。
- **バックエンド切替**: `Settings.generation_runtime`（podman | oci-ci）。`build_frontend` の
  検証・deadline（15 分ハードキャップ）・再試行の枠は共通。podman 経路は無変更（回帰なし）。
- **reconcile（ADR §4）**: API 起動時 + 5 分周期。孤児 `jetuse-builder-*` CI（N1+3 分超過）の
  補償削除・N1 超過 provisioning の failed 化・残置ジョブオブジェクトの掃除。
- **infra**: dev-app `gen.tf`（public OCIR repo 2 本・subnet/AD/イメージ URL/タイムアウトを
  API env へ merge — SP3-07 の main.tf とはファイル分離）。
- **イメージ**: `infra/images/gen-ci/`（Containerfile.gen / Containerfile.build + 相スクリプト +
  xfer.mjs）。`@ai-sdk/openai@4.0.9` を生成イメージへ事前導入（SP3-06 residual 解消 —
  `--network=none` で provider ロード成立を確認）。`ops/deploy-dev-app.sh gen-images`
  （apply 後に実行・既存タグ skip・`GEN_IMAGE_FORCE=1` で強制）を all/deploy-dev.yml に組込み。

## 2. 実機で発見し修正した事実（Tips）

1. **`container retrieve-logs` はコンテナ INACTIVE 後 409**（"Container is in : Inactive"）。
   SP3-07 Tips の retrieve-logs は ACTIVE コンテナ限定だった。→ N4 ログは**相スクリプト自身が
   trap EXIT で書込専用 PAR（LOG_URL）へ PUT** する方式へ変更（失敗時も opencode ログが
   `config.generation.error` に残る — 失敗系 E2E で実証）。retrieve-logs は起動失敗時のフォールバック。
2. **タグは不変**: 相スクリプトを直したらイメージタグを進める（oc1.17.15 → oc1.17.15-r2）。
   gen-images の「既存タグ skip」が成立する前提。
3. OpenCode の provider npm パッケージは `$HOME/.cache/opencode/packages/<name>@<ver>` に入る。
   イメージビルド時に到達不能 baseURL でダミー実行 → 導入だけ済ませ、`test -d` で fail-closed 確認。
4. 120b の最小プラン生成は**速い**: 15 秒タイムアウト注入でも再試行が完走する例を観測
   （opencode 単体 <15s）。失敗系検証の強制タイムアウトは 5 秒が確実。
5. PAR の実挙動: 読取 PAR への PUT・書込 PAR への GET はとも **404**(存在秘匿型の拒否)。
   また **PAR は期限切れ後もリソースとして残る**(59 件残存を実測 — review-2 M001)。
   ジョブ finally での削除 + reconcile の期限切れ掃除で回収する。
6. **ローカルからの再デプロイは同一 SHA だと API CI が置き換わらない**: image tag =
   `<branch>-<short-sha>` のため、未コミットの反復では同タグ上書き push になり RM 変数が
   不変 = terraform no-op。`GITHUB_REF_NAME=<別名> ops/deploy-dev-app.sh all` でタグを変えて
   強制置換する(GitHub CI は push ごとに SHA が変わるため影響なし)。
