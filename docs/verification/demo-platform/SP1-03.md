## 実装要約

- `service/routes/demos.py` 新設: `APIRouter(prefix="/api/demos/{demo_id}")`。全ルートが
  SP1-02 seam で `DemoContext` を取得し、その箱(`demo_<id>` 名前空間)だけを操作する。
  - 閲覧・実行(chat / GET rag/files)= `require_demo`(公開デモは非所有者も可)。
  - **書き込み(POST/DELETE rag/files)= `require_demo_owner`(所有者のみ。usecases の
    「公開は取得・実行可、編集・削除は所有者のみ」と同じ規則)**。
  - demo chat への `conversation_id` 持ち込みは 422 で拒否(会話の demo_id 紐付けは SP2)。
- ハンドラ本体は既存ルートと共有(重複なし):
  - chat: `routes/chat.py` の本体を `stream_chat_response(req, user, rag_ns)` に抽出。
    `rag_ns` は RAG 文書の名前空間キー(user 単位= `user.subject`、デモ= `ctx.namespace`)。
    差し替わるのは `rag.get_store_id` / `rag.resolve_citation_filenames` /
    select_ai・opensearch `generate` の第一引数キーのみ。**監査ログ・会話・エージェント/MCP
    解決は実ユーザーのまま**(監査証跡は人に紐づける)。コア関数のシグネチャ変更なし。
  - rag: `routes/rag.py` を `list_files_response / upload_file_response / delete_file_response`
    に整理し、検証(拡張子/サイズ/空)は両ルートで同一。
- `jetuse_core/rag.py add_file`: CP→DP 伝播ラグ404(SPIKE-03 既知)を**有界リトライ(最大6回)**で
  吸収。デモは箱ごとに新規ストアのため初回 upload が通常経路になることへの対応。
- SP1-01 ディスクリプタ `chat` / `rag.search` の routes にデモスコープルートを追記
  (routes 実在テスト `test_capabilities.py` が乖離を検出する構造は維持)。
