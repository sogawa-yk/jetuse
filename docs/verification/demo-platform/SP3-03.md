## 検証 1: LLM 認証経路（§4.4-2 — 技術リスクの核）→ **B1 署名プロキシ成立**

- `spikes/sp3_03_sign_proxy.py`: `oci-genai-auth` の `OciUserPrincipalAuth`（httpx.Auth）を
  `httpx.AsyncClient(auth=...)` に載せ、大阪 GenAI へ IAM 署名 + CompartmentId を注入するリバースプロキシ。
  クライアントの `Authorization: Bearer dummy` 等 OCI スコープ/署名/認可ヘッダは**転送せずサーバ側で固定**。
- **v2（成立確認）実測（証跡 spike-v2-auth-path.log / model-probe.log）**:
  - 非 stream chat.completions（gpt-oss-120b）: **HTTP 200 / 0.27 秒**。
  - SSE stream 成立。応答は `aiter_bytes()` でデコード転送（上流が identity を無視しても壊れない — R9-F009）。
  - OpenCode（openai-compatible provider, `baseURL=プロキシ` + ダミー key）から
    headless `opencode run` → write ツール呼び出し → ファイル生成まで **2.4 秒で完走**。
- **v3（最小権限化・R9-F001/R10 で硬化。証跡 spike-v3-restricted-proxy.log）**: 転送先を
  `POST /chat/completions` の**完全一致 allowlist** に限定し実機再検証 — 横移動（files/responses/
  vector_stores/conversations）= **403**、非 allowlist モデル = 403、GET = 405、Content-Type≠json = **415**
  （R10-F007）、Content-Length 超過 = **413**（本文全読み前に打ち切り — R10-F002）、既定 allowlist は
  実証済み **120b のみ**（R10-F006。20b は env で明示追加時のみ）。正常系（chat=200 / SSE）は維持。
- **結論: 「oci-genai-auth は OpenCode に注入不可」は署名プロキシで解け、かつ chat.completions への
  最小権限で成立**。生成プロセスは鍵レス（S2）。実装時はリソースプリンシパル
  （`AUTH_MODE=resource_principal` — 既存 `_signer()` 分岐）に差し替わるだけ。

## 検証 2: OpenCode headless 実挙動・生成品質（§4.4-6）

| 実験 | モデル | 結果 | 生成時間 | ピーク RSS |
|---|---|---|---|---|
| gen1: プラン→SPA（ホスト直接） | openai.gpt-oss-120b | **成功**（3 ブロック実装・build 緑） | **24 秒** | 614MB |
| gen2: 同（モデル切替 F2） | google.gemini-2.5-flash | **失敗**（15 分 timeout・成果ゼロ。chat.completions 200 は返るが agentic ループが進行しない） | 15 分打ち切り | 346MB |
| gen3: 同（コンテナ内・DNS 遮断・2cpu/2GB/pids512・鍵レス） | openai.gpt-oss-120b | **成功**（build 緑・静的検査違反ゼロ） | **3 分 26 秒** | —（コンテナ内） |

- F2（モデル切替）は **opencode.json の provider 宣言 + `model` の差し替えだけで成立**
  （gen2/gen3 は同一プラン・同一プロンプトで config のみ変更）。切り替え自体は機構として動くが、
  **モデルにより agentic 完走性が大きく異なる**のが実測の核心（下記）。

- 入力 = 固定スキャフォールド（`spikes/sp3_03_scaffold/` — Vite+React+固定 API クライアント+AGENTS.md）
  + specs/19 §3.2 例のデモプラン（設備保全・3 ブロック）。
- gen1 生成物の検査: 3 ブロック（rag.search / dbchat / chat）とも日本語 UI・suggested_prompts チップ・
  SSE ストリーミング表示を実装。**生 fetch / 絶対 URL / プロトコル相対 = 検出ゼロ**（固定クライアント
  経由のみ — S3(a) の誘導が機能）。`npm run build` を OpenCode 自身が実行し緑を確認して終了。
  ただし gen1 は**旧 client.js（契約不整合）**時点の生成であり、chat に `{system_prompt}`（現行は
  `systemPrompt`）を渡し dbChat の戻り値を表示しない実行時バグを含む（下記 findings）。
- **現行スキャフォールドでの再生成（codex-review F003 対応）= gen5**: review-2 修正後の
  client.js/AGENTS.md で再生成すると、**現行シグネチャどおり**に生成される（`await chat(msgs,
  onDelta, {systemPrompt})`・`await dbChat(q, onSql)` の戻り値 `{sql, rows}` を表・`try/catch` で
  各ブロックのエラー表示）。静的検査違反ゼロ・build 緑（証跡 gen5-contract-check.log /
  gen5-App.jsx.sample）。ここで実証できたのは**API シグネチャの契約整合と build 成功まで**。
- **生成品質は依然不安定（codex-review F005 — 「stale-state 解消」は撤回）**: client.js を戻り値方式に
  変えたことで gen1 の「古い closure 変数を append」型バグは消えたが、**gen5 は別種の state バグを
  独立に混入**した — chat ハンドラが `setMessages(newMessages)`（末尾=user 発話）の直後、初回 delta で
  `prev.slice(0, -1)` により**その user 発話を削除**して assistant で置換するため、次ターンの履歴から
  利用者発話が欠落する（`chat()` の戻り値 `full` も未使用）。**さらに gen6 も同種の実行時バグ**
  （codex-review R11-F008）: chat() の戻り値を使わず stale な chatStreaming を履歴へ追加し応答が空になる
  （AGENTS.md の明示指示にも不従）。**build 成功率と実行時契約成功率を分けて数えると、build/静的検査は
  gen1/5/6 とも通る（3/3）が実行時契約の成功例はゼロ（0/3）**。→「build 成功 ≠ 実行時正しさ」を裏づけ、
  生成品質は本質的に不安定（モデルが state 管理を誤る）。ADR-0023 の緩和（ブロック UI 部品の焼き込み・
  人間のプレビュー確認ゲート・安価な再生成）を裏づける。実ブラウザ/契約モックでの実行時検証
  （複数ターンの送信履歴・RAG delta・NL2SQL→execute→表・SSE error/切断）は実装フェーズの受け入れ条件
  （配信経路が ADR 承認後に生まれるため本フェーズ不能）。**「動く SPA」は固定 UI 部品または複数ターン
  契約モックが通るまで成功例に数えない**（本フェーズの実証は build 成功 + 契約整合の静的確認まで）。
- **固定クライアントの契約整合（codex-review F002 対応）**: スパイク初版の client.js は実 API 契約と
  乖離していた（ChatRequest の `model` 必須欠落・`system_prompt` フィールド誤り・dbchat の
  nl2sql→execute 二段欠落）。実コード（service/schemas.py・routes/demos.py・routes/dbchat.py・
  routes/chat.py の SSE フレーム形式）に合わせて修正済み（model 注入はビルド時定数
  `import.meta.env.VITE_DEMO_MODEL`（CSP `default-src 'self'` 下でインライン script 不可のため —
  R9-F008）+ 既定キー、BASE は `/app` 境界から導出（R10-F004）、system_prompt→system メッセージ化、
  `data:{"sql"}` → `POST /dbchat/execute`、
  `data:{"error"}` の例外化・破損フレーム非黙殺・**[DONE] 未受信 EOF の例外化 + reader の
  finally 解放（review-2 F003）・認証注入 seam `setAuth()`（review-2 F002 — AUTH_REQUIRED=true の
  成立条件は ADR §3.5）**）。修正後も `--network=none` ビルド緑（offline-build.log 追記）。
  **実ルートへの疎通契約テストは実装フェーズの受け入れ条件**（プレビュー E2E — 本フェーズは
  ADR 承認前のため実装・配信経路が存在しない）。
- **品質の限界（findings）**: gen1 の ChatBlock に React の stale-state バグ（応答完了時に古い
  `stream` 変数を履歴へ append → 空文字が入る）。**ビルド成功 ≠ 実行時正しさ**。緩和策は ADR §リスクに
  記載（ブロック実装をスキャフォールド側へ焼き込み、生成は「プラン→構成の配線」に限定する等）。
- **headless の落とし穴（Tips 収録）**: `opencode run` は permission 未設定だと承認プロンプトで
  **無限ブロック**する（タイムアウトまで無応答）。headless では `permission: {edit/bash: allow,
  webfetch: deny}` を config で明示するのが必須。
- モデル対応状況（署名プロキシ経由の実測 — 各モデルのコマンド・HTTP status・時刻・応答要約は
  証跡 `model-probe.log`）:
  - chat.completions 疎通**可**（HTTP 200）: openai.gpt-oss-120b（0.27s）/ gpt-oss-20b（0.08s）/

    **不可**: cohere.command-a-03-2025 → `HTTP 400 Unsupported OpenAI operation`（chat.completions 自体が

  - OpenCode の**生成完走**まで実証できたのは **openai.gpt-oss-120b のみ**（2/2）。
    gemini-2.5-flash は疎通可だが完走せず（gen2）。gpt-oss-20b / llama-3.3-70b は疎通のみ確認

## 検証 3: egress / オフラインビルド（§4.4-3）→ **C1 成立**

- スキャフォールドの依存（node_modules 41MB。package-lock.json で固定: vite 6.4.3 / react 18.3.1 —
  lockfile はスキャフォールドに同梱）を焼き込み、`podman run --network=none` で `npm run build` →
  **成功（exit 0。専用証跡 `offline-build.log` — コマンド・イメージ digest・全出力・生成物検査込み）**。
  npm レジストリへの egress 不要。
- **findings（S3(b) 設計に影響）**: dist の絶対 URL 走査で React 同梱の定数 URL
  （reactjs.org error-decoder / w3.org XML namespace）が検出される。素朴な「絶対 URL = 不合格」は
  クリーンビルドさえ落とす → 静的検査はクリーンビルド由来の既知定数を許容するベースライン方式が
  必要（ADR-0023 リスク節）。
- **findings（規律の強制方法）**: gen3 で OpenCode が AGENTS.md の「npm install 禁止」に反して
  `npm install` を実行（オフラインのため実害なし）。禁止はプロンプト任せにせず環境で強制する
  （egress 遮断 + 生成後の lockfile/package.json 改変検査 — ADR-0023 リスク節）。
- **信頼ビルド相のマウント検証（codex-review F003）= trusted-build.log**: **read-only root +
  RO node_modules**（`podman run --read-only -v $PWD:/work:ro`）で、vite の一時ディレクトリ
  （`.vite-temp` / `.vite`）と出力（`/out` = tmpfs）だけを RW にして `vite build` → **成功
  （✓ built in 577ms・dist 生成）**。負の系: 同マウントで `src/api/client.js`・`package-lock.json`・
  `node_modules/*` への書き込みは**すべて EROFS 拒否**。→ ADR §1 の「ツールチェーンだけ信頼・入力は
  敵性・ビルド相も egress ゼロ/RO root/限定 scratch」が実機で成立。
- **生成相が src のみ生成（codex-review F001）= gen6**: AGENTS.md を「`src/` のみ編集・`npm run build`
  しない」に更新して再生成すると、**dist 未作成（src/ のみ）・保護ファイル無改変**で完走（証跡
  gen6-buildfree.log）。ADR §1 の 2 相分離（生成相は src 差分のみ・ビルドは信頼相）が生成側でも成立。
- gen3（DNS 遮断コンテナ）で OpenCode 本体も外部到達なしで動くかを検証（下記）。

## 検証 4: ランタイム A/B/C（§4.4-1）

- **前提の実態確認**: jetuse:dev に OKE クラスタは**存在しない**（2026-07-05 のリセットで破棄済み。
  旧 kubeconfig の API エンドポイントはタイムアウト）。現行のデプロイ実体は RM スタック
  `jetuse-dev-app` の **Container Instance**（CI.Standard.E4.Flex 1ocpu/4GB, subnet
  jetuse-dev-app-private）。→ 比較ドキュメントの「A2=OKE Job」は現環境では成立せず、
  **B の実体 = 生成ごとの Container Instance 起動**に読み替えて実測。
- podman（コンテナ境界の隔離・上限の成立確認）= gen3:
  - `--memory=2g --cpus=2 --pids-limit=512 --network=slirp4netns:allow_host_loopback=true
    --dns=<blackhole>`・OCI 資格情報の非マウント（鍵レス）・baseURL=ホスト上のプロキシのみ、で
    OpenCode の生成 → `npm run build` 緑まで**完走（3 分 26 秒）**。
  - **正直な限定（codex-review F004）**: この構成が実証するのは「**DNS 名前解決なしでも生成が完走する**」
    ことに限られる。blackhole DNS は名前解決を止めるだけで、外部の直接 IP・許可済みホストの別ポート・
    メタデータ/RPST エンドポイントへの接続は遮断していない。よって「egress=プロキシのみ」「S1 の egress
    絞り込み成立」の**裏付けにはならない**。egress の実封じ込めは NSG/セキュリティリスト（実 CI）の
    仕事であり、**「外部ホスト名・外部直 IP・メタデータ/RPST・API ホスト別ポートが失敗する負の E2E」を
    実装受け入れ条件**とする（ADR-0023 決定 1・3）。
  - コンテナ起動オーバーヘッド: node:22-slim で **約 0.7 秒**（warm image）。
- Container Instance 起動レイテンシ（実測 — `jetuse-spike-opencode-b`、計測後削除・DELETED 確認済み）:
  **CREATING → ACTIVE 43 秒**（docker.io の node:22-slim 233MB / CI.Standard.E4.Flex 1ocpu/4GB /
  大阪 AD-1 / 証跡 spike-ci-latency.log）。
- A（API コンテナ内 subprocess）は動作はするが（gen1 がその形）、S1/N7 を構造で満たせない点は
  比較ドキュメントの整理どおり。プレビュー API の CI が 1 OCPU/4GB であることを実機確認しており、
  生成（RSS 614MB + ビルド CPU）を同居させる余地も乏しい。→ ADR は B'（生成ごとの CI）を採用案とする。
- **findings（S2 の落とし穴 — codex-review F001）**: 生成 CI を素朴に jetuse:dev へ作ると既存 DG
  `jetuse-internal-dg` の matching rule に入り、リソースプリンシパルで広権限を得うる（podman 検証
  では再現できない実 CI 固有の経路）。ADR は「生成 CI を DG 対象外に配置（IAM 人間ゲート）+
  実装受け入れに RP 不使用の負の E2E」を決定に含めた。なお本エージェントに IAM 読取権限はなく
  （`oci iam dynamic-group list` = 404 NotAuthorized 実測）、matching rule の現状確認自体が
  人間ゲート事項。

## 検証 5: N1/N7 実測（§4.4-5）

- **提案シェイプ相当の複数回計測（codex-review F003 対応）**: podman `--cpus=1 --memory=4g
  --pids-limit=512` + 鍵レス + **DNS 遮断**（名前解決なし — 実 egress 封じ込めは実 CI の NSG が担う。
  外部直 IP/メタデータ/別ポートは本近似では遮断していない）で同一プラン生成を 3 回:
  **95s / 165s / 96s（中央値 96 秒・3/3 で build 緑・静的検査違反ゼロ — 証跡 gen4-n1n7.log）**。
- N1: CI 起動 43 秒（実測）+ 生成中央値 96 秒 + データ投入（RAG store 初回 40〜60 秒 — SP1-03 実測）
  = 典型 3 分前後 → **p50 ≤ 5 分・ハードタイムアウト 15 分（spec 既定値を暫定確定）**。
- N7（CPU/メモリ）: 1cpu/4GB/pids512 で 3/3 完走・ピーク RSS 614MB（ホスト実測）→ **1 OCPU / 4GB**
  暫定確定。
- **N7（ディスク）は未検証（codex-review F005）**: 実測は使用量の小ささ（書き込み 62MB / OpenCode
  状態 6.6MB / dist 152KB）を示すだけで、2GB 上限の**強制方式と超過時 failed は裏づけていない**。
  実 CI の quota/専用ボリューム/監視 kill と境界挙動の実測を実装受け入れ条件とする。
- **限定**: podman 近似であり実 CI 上の生成ではない。**実 CI での分布計測（CI 作成→成果物受領）を
  実装受け入れ条件**として ADR-0023 §決定 5 に明記。

## 総括

- **技術リスクの核（LLM 認証経路 = 署名プロキシ）は解けた**。egress オフラインビルド・鍵レス生成・
  2 相分離（生成/信頼ビルドの隔離）・生成品質の実挙動は実機で確認し、ADR-0023 を Proposed で起草済み。
- **本フェーズで解けていない・承認をゲートする未解決点が 3 点ある（ADR-0023 リスク節と同一）**:
  1. **生成 CI のネットワーク隔離（OCIR pull/egress 分離 + ブローカー迂回防止 + AUTH_REQUIRED）**
     （codex-review R7-F002 / R11-F001）: pull 成功後にコンテナから OCIR/他サービス/直 IP/RPST が叩けず、
     生成プロセスがブローカーを迂回して通常 API へ直結できず、生成 CI は `AUTH_REQUIRED=true`
     （jetuse-dev の `AUTH_REQUIRED=false` = 無条件 dev-user では横移動可）— これらは podman 近似では検証
     できず、実 private OCIR イメージ + 想定 NSG の**実 CI 負 E2E が必要**。ネットワーク/NSG/IAM に踏み込む
     ため人間ゲートであり、**成立しなければランタイム B' の再検討**（例: OKE 復帰 + NetworkPolicy）を要する。
  2. **S3 の完全な fail-closed は「自由 JSX 生成」と両立しない + 生成 SPA の同一オリジン相互隔離**
     （codex-review R8-F002 / R13-F001）: (a) CSP は navigation 経由の外部送信を止められない。(b) 同一オリジンに
     全デモを載せるとデモ A の生成 JS がデモ B の能力 API を叩ける（path 別 Cookie/固定ヘッダをブラウザ/生成 JS が
     満たす）。真の隔離には**宣言的生成 + デモごとの別オリジン（または opaque-origin sandbox）**が要る。
     specs/19 §4.3 脅威モデルは完全封じ込めを求めていない（権限昇格なし・CSP 主）ため、**「自由 JSX +
     CSP/静的検査 + 同一オリジン」で SP3 を進めるか、宣言的生成 + 別オリジン隔離へ倒すか**の判断を要する。
  3. **N7（プロセス数・ディスク上限）の強制方式が OCI Container Instance で成立するか**（codex-review
     R11-F004）: CI の resource_config は CPU/メモリのみ。**強制不能ならランタイム B' 自体が成立しない**
     ため、実 CI で PID/2GB 境界内成功・超過時 failed + CI 削除を承認前に検証するか、不可能なら OKE +
     cgroup/quota を判断対象にする。
- したがって**残る人間ゲート = adr_approval**（上記 3 点の可否判断を含む）**に加え、ネットワーク隔離の
  実 CI 検証・ネットワーク/IAM 判断**（DG matching rule 変更・`is_resource_principal_disabled` 前提の
  負の E2E を含む）。承認まで本実装に着手しない。
- 生成品質の限界（ビルド成功 ≠ 実行時正しさ。gen1/gen5 の 2 種の state バグ）と、モデル間の agentic
  完走性の差（gpt-oss-120b のみフル実証）は ADR のリスク節に記載し、実装時の設計（スキャフォールドへの
  ブロック部品焼き込み・宣言的 UI 化・allowlist 既定の絞り込み）に反映する。
