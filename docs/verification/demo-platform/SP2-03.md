## 1. 実装サマリ

主要 3 系統（chat / rag / dbchat）を縦に揃える最終ピース。既存ハンドラを namespace 引数の
共有内部関数へ整理し、デモスコープルートを SP1-03 の rag と同型で敷いた。

### 1.1 会話のデモ紐付け（§4.2）
- `POST /api/demos/{id}/conversations`（require_demo）を追加。行は
  `owner_sub = owner_key(user.subject)`・`demo_id = ctx.demo_id`。排他リース下で INSERT。
- demo chat の conversation_id 422 拒否を解除。指定時は
  `owner_sub = owner_key(user.subject) AND demo_id = ctx.demo_id` を検証（不一致 404）。
- **demo 会話は OCI Conversation を作らない**（LTM のデモ間混線・外部会話孤児を構造排除）。
  継続契約 = クライアントが messages に全履歴を再送。
- **user 単位の会話 API は全 verb（一覧/GET/DELETE/title/set_oci/chat 継続）で `demo_id IS NULL`
  を強制**（両方向の持ち込みが 404。既存データは全行 NULL のため Public 挙動不変）。

### 1.2 dbchat 縦切り（§4.3）
- デモスコープ dbchat = datasets ターゲット固定: nl2sql / execute / schema + datasets CRUD。
  読み取り = require_demo、書き込み = require_demo_owner。dbchat.py を `*_response` 共有本体に整理。
- demo nl2sql のモデルは **config.dbchat.model 固定**（リクエストの model 入力は無視 —
  非所有者が共有プロファイルを再構築させない）。

### 1.3 層2 = fail-closed の SQL ゲート（§4.3。codex review-1 B001 で allowlist 化）
`jetuse_shared.sqlguard.enforce_sql_boundary` を **allowlist 方式**で実装:
FROM/JOIN のテーブル参照を DUAL・WITH の CTE・`SH.<表>`（修飾必須）・呼び出し元の登録済み DS 表
（素名 or app スキーマ修飾）だけ許可し、それ以外（未知 synonym・別スキーマ・辞書ビュー・
table function）を**一律拒否**。位置非依存でパッケージ（DBMS_/UTL_）・スキーマ修飾の関数呼び出し・
SYS./SYSTEM. 修飾・@ DBリンク・判定不能（q''）も拒否。EXTRACT(.. FROM ..) 等の関数内 FROM は
括弧文脈で句境界と区別。`execute_readonly` の全呼び出し元（user/デモ execute・preview_table・
datasets.preview・agents `query_database`・Fn ルーター・agent-containers `agent_db.py`）を移行。
agent 経路は **owner なしモード**（JETUSE_DS_/辞書/パッケージ全拒否・SH 修飾照会のみ通過）。

多層構成: **行データの境界 = 層1（VPD / DBMS_RLS、SP2-02 所有）**、**辞書・パッケージ面 = 層2**。
