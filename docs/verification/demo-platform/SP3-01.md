## 1. 実装

### builder_sessions エンティティ（specs/19 §2.1）
- 単文 migration 2 ファイル: `025_builder_sessions.sql`（CREATE TABLE — status は
  `hearing`/`designed` の 2 状態 CHECK・transcript/requirements/plan は IS JSON）・
  `026_builder_sessions_idx.sql`（`idx_bs_owner(owner_sub, updated_at)`）。
- ランナー事後条件検証の対象化: `migrate.py _EXPECTED_POST` に 025/026 の期待形
  （列型・既定値・CHECK・索引列）を追加。
- リポジトリ `jetuse_core/builder_sessions.py`: demos の流儀 — **所有者強制は SQL の
  WHERE 句**（0 行 = 404 存在秘匿）。owner_sub は識別列（raw sub。owner_key は通さない）。
  `save_hearing_turn` は `WHERE ... AND demo_id IS NULL` で「生成開始後は読み取り専用」を
  UPDATE 自体に内蔵（読み→LLM→書きの競合も 0 行 → 409 に収束）。
- 生成以降の進行は demo_id と Demo.status から導出（状態の二重管理なし）。
  SessionOut の `demo_status` は demos への LEFT JOIN で添付。

### ヒアリング API 3 ルート（specs/19 §2.4 — `service/routes/builder.py`）
- `POST /api/builder/sessions`（Body なし・hearing/transcript=[] で INSERT）/
  `POST .../{sid}/messages`（LLM 1 呼び出しの同期 JSON）/ `GET .../{sid}`。すべて require_user。
- messages 応答 = `{"reply", "requirements", "sufficient", "missing"}`。
- LLM 使用は usage_log に owner 紐づけで記録（specs/19 §8.3。ベストエフォート）。

### 構造化出力と「設計に足りる」判定（specs/19 §2.2・§2.3 — `jetuse_core/builder_hearing.py`）
- 既存 chat 基盤の既定モデル（`DEFAULT_MODEL` = gpt-oss-120b）を temperature 0 で流用。
  出力はコードフェンス除去 + `json.loads` + pydantic 検証（docunderstand の頑健 parse の流儀）。
  形式不正は**1 回だけ**再試行（形式エラーをフィードバック）、なお不正なら 502（fail-closed。
  失敗ターンは transcript に残さない）。
- **サーバ側の決定的再検査を最終判定**とする: LLM が sufficient=true でも必須
  （industry / use_case / data_profile のどちらか一方以上）が空・空白なら false に落とし
  missing に必須項目を合流。逆（必須充足なのに LLM が false）は LLM に従う。

### 信頼境界の入力上限（specs/19 §2.1）
- 発話 1 件 ≤ 4,000 文字（pydantic `max_length` — 超過/空は 422）。
- 50 往復 or transcript 直列化 256KB 到達で 422（detail に「新しいセッションを開始して
  ください」）。**LLM 呼び出し前に遮断**（LLM 入力の有界化を兼ねる）。

### ready ゲートの一般化（specs/19 §8.1）
- `require_ready_demo` を新設し、デモスコープ能力ルート router
  （`/api/demos/{demo_id}` prefix — chat / conversations / dbchat / db/datasets / rag 全 13 本）の
  **router 共通依存**として付与。`status='ready'` 以外（provisioning/failed/deleting）は
  存在秘匿と同じ 404。SP3-03 の `/app/` 配信も同 router に載るだけでゲートされる。
- demos CRUD メタ（GET/PATCH/DELETE）は従来どおり（所有者は非 ready でも status を見られる）。
- SP2 までに非 ready の demo を作る経路はないため既存挙動の変化なし = 既存スイート全緑 +
  `test_ready_gate.py` で回帰担保。
