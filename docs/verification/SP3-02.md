### デモプラン plan_version=1 スキーマ + fail-closed 検証（§3.2・§3.3 — `jetuse_core/builder_design.py`）
- pydantic **strict / extra=forbid**（型強制なし・未知フィールド不合格）。
  **配線はブロック型で固定**: `blocks[].type` = capability id のみ。URL・パス・HTTP の
  自由記述フィールドはスキーマに存在せず、extra=forbid が構造的に排除する。
- 検証項目（§3.3 の表どおり。上限は module 定数 = 仕様の既定値）:
  plan_version==1 / capabilities は語彙の部分集合（空・重複・語彙外 422）/
  screens 1〜5・blocks 1〜8・`blocks[].type ∈ plan.capabilities` /
  tables 0〜5（列 1〜20・rows 1〜500・表名重複禁止）/ 識別子 `^[a-z][a-z0-9_]{0,29}$` /
  型許可リスト `VARCHAR2(n CHAR)`(n≤1000)・`NUMBER`・`NUMBER(p[,s])`(p≤38,s≤38)・`DATE`・
  `TIMESTAMP`（**fullmatch** — 末尾改行を受理する `$` アンカーの罠を排除 = review-1 F005）/
  documents 0〜10（filename `^[a-z0-9_-]{1,64}\.(md|txt)$`）/
  能力⇔データ整合（`DATA_REQUIREMENTS = {dbchat: tables, rag.search: documents}` の**両方向**）/
  文字列長（title≤200・description/outline≤1000・system_prompt≤4000・suggested_prompts
  各≤200・5 件）/ 直列化 ≤256KB。
- screens[].id にも識別子制約・重複禁止を適用（§3.3 の字面を超える保守的検証 — 生成コードの
  ルーティング/命名に使われる信頼境界のため。コメントで根拠明記）。
- 語彙は**検証コンテキスト必須**（コンテキストなしの検証は不合格 = fail-closed）。
- 合格プランは `model_dump(exclude_none=True)` の正規化 dict として保存（再検証べき等）。

### LLM 生成の頑健化（実機で発見した欠陥 2 件の修正 — deploy.log 追補2・3）
- 実機の gpt-oss-120b は temperature 0 でも出力が揺れ、(a) 完全な JSON の直後に余分な
  データが続く（`Extra data`）、(b) **構造的に壊れた JSON**（オブジェクト間に余分な閉じ括弧 —
  `Expecting ',' delimiter`）を返すことがある（同一イメージの podman 隔離再現で確定・
  失敗形を fixture 化）。(b) は抽出側で回復不能で、codex review-1 のライブ E2E では
  design 連続 422（blocker F001）となった。
- 対策（三層。受け入れ判定は strict スキーマ検証のままで fail-closed 不変）:
  1. **json_schema 構造化出力**: Responses API の `text.format=json_schema`
     （`DemoPlan.model_json_schema()`・strict）で生成を依頼。大阪プレビュー実機で受理・
     **6/6 合格・出力ほぼ決定的化**を確認して採用（chat API 系モデルへの適用は上流対応
     未確認のため行わない — 受け皿は下記 2・3）。
  2. **頑健抽出**: fence 除去 → 最初の `{` から `json.JSONDecoder().raw_decode` で最初の
     オブジェクトのみ取得（前置き・後続の余分データを捨てる）。構文エラーは位置+周辺付きで
     再生成フィードバックへ。parse 失敗の生出力（有界）は logger.warning で観測可能化。
  3. **有界再生成**: 最大 2 回（§3.1）。

### プラン語彙の構造的導出（§3.4 — `jetuse_core/capabilities.py`）
- `demo_plan_vocabulary()` = `demo_safe=true` かつ `routes` に `/api/demos/{demo_id}/` 配下の
  パスを 1 つ以上持つ能力。**chat / rag.search / dbchat をハードコードしない** — 合成カタログの
  単体テストで導出の構造性を証明（デモスコープルート追加だけで語彙が自動拡張）。
- カタログ生成は `build_catalog()` を `GET /api/capabilities` と design が**共用**
  （§3.1「同じ生成関数を内部呼び出し」。単一真実源 = CAPABILITIES 登録簿）。
- LLM プロンプトには語彙フィルタ済みカタログの記述部分（capability / summary / when_to_use）
  だけを入れ、LLM 入力を有界化。プロンプトは固定・`PROMPT_VERSION = 1`（§3.1 の版数管理）。
