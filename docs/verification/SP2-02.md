## 2. 実装（specs/18 準拠）

- **予約 ledger**（`rag_file_ledger`）: quota 行 FOR UPDATE → 件数検査 → pending INSERT の予約 Tx。
  箱上限は **ledger 行数**（`count_for_owner`）で測る（外部削除失敗で残した pending も数える）。
  外部作成はすべて予約の後（テナンシ枠の空消費を防止）。
- **完全ハッシュ命名**: OCI File 名・原本名・dataset 表名に `file_key`（`<sha1(owner)>/<rid>.<ext>`）/
  完全 sha1 を使用（衝突・越境防止）。
- **排他リース**（`demo_lease.py`）: `JETUSE_LOCK` cover package（`ORA_HASH(demo_id)` で決定的ロック ID
  → `DBMS_LOCK.REQUEST` X_MODE / `DBMS_LOCK.RELEASE`）+ 専用小プール + 再入トークン。demo 単位の
  作成/削除を直列化。
- **VPD 基盤**（`vpd.py`）: アプリ context + setter パッケージ + ポリシー関数 + `JETUSE_DS_*` への
  `DBMS_RLS.ADD_POLICY`。起動時 `verify_integrity` で GRANT 済み表のポリシー欠落を検出（fail-closed）。
- **DELETE 後始末**（`demo_cleanup.py`）: datasets→files→store→ledger/孤児→select_ai→opensearch→原本→
  行、を段階化。進捗永続で再 DELETE が途中から収束。会話チャンクも削除。
- **fail-closed アップロード gate**（DB 永続 `rag_file_quota`）: 既定 `'N'`。bootstrap の reconcile が
  未管理 File ゼロを確認して初めて `'Y'`。bootstrap は uvicorn と別プロセスのため DB 行で状態共有。
  さらに **起動世代 `gate_boot_id`**（entrypoint.sh が両プロセスへ共有 `APP_BOOT_ID` を export）で
  前回起動の 'Y' を無効化し、reconcile 完了前の受付窓を塞ぐ（review-8 B001）。
- **既存 File の grandfathering**（アップグレード互換）: SP2-02 導入前の `rag_files` は ledger へ
  backfill せず、reconcile が oci_file_id を未管理判定から除外し、上限は ledger + grandfather で
  数える。backfill 方式（adopted 行の推測 locator が個別 delete/cleanup で孤児化する経路）を廃した
  （review-11 B001。scenario-1 A1f/A1g）。
- **構成ドリフト耐性**: 個別 delete・箱 DELETE は行ごとの write-ahead locator で旧 project/endpoint の
  File・原本・OpenSearch チャンクを辿る。demo_cleanup は台帳（demo_backend_targets）を正とし、
  現在設定は記録の無い legacy だけのフォールバック（review-7 B004 / review-8 B002）。
- **後方互換ハードニング**（review-12）: (B001) datasets の CREATE TABLE 失敗（ORA-00955 名前衝突を
  含む）は `created_table` ガードで**自作の表のみ DROP** し、他データセットの実表を消さない（データ
  損失防止）。(B002) 原本 Object Storage prefix と Select AI 索引 location の owner セグメントを
  **demo=完全 sha1 / user=main 互換の raw owner** に分岐し、既存 user の 8hex 索引が更新後の新規
  アップロードを取り込めるようにした（OCI Files 名は id 参照ゆえ完全 sha1 のまま。scenario-1 F2/F3）。
  (B003) chat/agent の RAG 検索 store 解決も `rag.resolve_store_for_read` 経由で `owner_key_gate` を
  通し、write/list と同じ fail-closed 一貫性で移行窓の越境参照を塞ぐ（正常配備は no-op）。
- **排他リースの VPD 非依存化**（review-13 B001）: `JETUSE_LOCK` cover package を `lock_definitions()`
  として VPD 固有定義から分離し、`reapply_definitions` が **VPD 無効でも lock を作る**。旧実装は排他
  リース定義を VPD 定義と束ね `vpd_enabled` でしか作らず、DBMS_LOCK を付与した Public/既定デプロイでも
  demo の作成・更新・削除が `jetuse_lock` 不在で恒久 503 になっていた（scenario-5 S9 で Public の
  lease 取得を実機確認）。あわせて chat の Select AI/OpenSearch generate 経路も `owner_key_gate` を
  通す（review-13 M007）。
- **原本 Object Storage の cleanup 健全性**（review-14）: (B001) 原本バケットは `versioning=Disabled`
  を `_put_original` 前段で fail-closed 確認（Enabled/Suspended だと delete_object が旧 version を残し
  回収不能）。(B002) PUT・個別/箱 DELETE・locator が `_resolve_os_namespace`（config 優先・無ければ
  `get_namespace`）で同一 namespace を使う（旧: PUT が実値・locator が設定値で drift/空時に別
  namespace を走査し孤児化）。scenario-1 A3 で real bucket=Disabled・ns 一致を実機確認。

## 3. E2E シナリオと結果（`runs/2026-07-06T1113_SP2-02/e2e/`）
