# ADR-0018: 認証付き MCP 登録の Vault 書込みモデル（同期＋補償 / outbox は非採用）

日付: 2026-06-29
状態: **提案（ドラフト）** — 人間レビュー待ち（BE-08 / loop 自走で起票。承認は人間ゲート）

## 背景

BE-08 で `POST /api/agent/mcp-servers` の認証付き登録（旧 501）を実体化した。実トークンは Vault に
`create_secret` で束ね、DB は OCID 参照のみ保持する（実値は本体のみ / ADR-0014）。実装は HTTP
リクエスト内で **同期的に** Vault 書込み→ACTIVE 待ち→DB INSERT を行い、失敗時は補償削除する。

codex レビュー（review-3 / BE08-R3-002・R3-003）で、より堅牢な代替として次が提案された:
- 作成・削除を **outbox / state-machine** で永続化し、バックグラウンドで成功まで再試行する。
- HTTP 再送を跨ぐ **クライアント提供の冪等キー**で、GW タイムアウト後の重複 secret/登録を防ぐ。
- 作成・補償を **非同期ジョブ**化して応答遅延と GW 60秒境界の競合を避ける。

## 決定（本タスクの範囲）

BE-08 では **同期＋補償（best-effort cleanup）モデルを採用**し、outbox / 非同期ジョブ /
HTTP 冪等キーは **本タスクでは非採用**（将来の独立タスクで再検討）。理由:

1. BE-08 の受け入れ条件は「501 解除・資格情報を Vault 保持・SSRF 維持・無認証経路非破壊・テスト/lint」。
   非同期基盤の新設はこの範囲を超え、新規の永続ストア・ワーカー・配送保証という別ドメインを持ち込む。
2. 実 Vault 書込みは **IAM 人間ゲート**で、現状この経路は本番で常時稼働していない（先回りの大規模
   ハードニングは投機的）。
3. 同期モデルでも、レビュー指摘の具体的な破綻は**個別に塞いだ**（下記「緩和策」）。残るのは
   「GW タイムアウト後にサーバ側が完走し、利用者の再送が新 secret を作る」希少ケースに限られる。

## 緩和策（同期モデル内で実施済み）

- ACTIVE 待ちに **monotonic な総 deadline（25秒）** を被せ、OCI クライアントは **timeout 明示
  (5,15)＋再試行無効**。create と待機の合計を API Gateway の 60秒より確実に下回らせる（R3-003）。
- 削除予約を **DB 行削除より前**に確定し、失敗時は行を残して 503（孤児防止 / R2-002）。
- 削除予約のエラーは安全側に分類（R3-002/R4-001/R5-002）: **404 は fail-closed**（OCI の
  NotAuthorizedOrNotFound が権限不足と不在を区別しないため、live secret を残さない）。**409 は
  `get_secret` で削除進行中（SCHEDULING/PENDING/DELETING/DELETED。CANCELLING_DELETION は除外）を
  確認できたときだけ**冪等成功として DB 行削除を継続。「予約成功後に DB 失敗→再送が詰まる」回避は
  この正当ケースに限定する。
- 削除時刻は受信から **2 日後**（OCI 下限 1 日の境界割れ回避 / R3-001）。
- 作成失敗時は **自管理 secret のみ補償削除**（外部管理 OCID は触らない）。`opc_retry_token` で
  create_secret 自体は冪等。

## 残リスク（人間判断・将来タスク）

- **HTTP 再送跨ぎの重複**: GW タイムアウト境界でサーバ側が完走した場合、利用者の再送は新 sid で
  別 secret を作り得る。完全な排除には HTTP 冪等キーか非同期ジョブ＋outbox が要る（本 ADR で非採用）。
- 採用するなら独立タスクで、配送保証・可観測性・クリーンアップ GC を含めて設計する。

## 影響

- 仕様外の実装判断のため本 ADR を起票。**承認は人間ゲート**（ループは自己承認しない）。承認されれば
  specs/16 にモデルを 1 段落追記する。却下/別方針なら本 ADR を改訂する。
