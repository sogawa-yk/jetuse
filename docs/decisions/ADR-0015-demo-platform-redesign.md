# ADR-0015: デモ生成プラットフォームの再設計（ADR-0013 を置換）

日付: 2026-07-06
状態: 承認（2026-07-06 施主承認）
関連: ADR-0013（置換対象）／specs/17-demo-platform-redesign.md／ADR-0016（ブランチ）

## 背景

2026-07-05 のリポジトリ方針転換で dev（EXB v2/RAG）を破棄し main のみへリセットした
（`memory/repo-reset-main-only.md`）。過去のデモ生成プラットフォーム設計（ADR-0013 / specs/16）の
実装本体は dev と共に失われ、L1 manifest（署名含む）だけが main に生存している。

改めて同じ構想（フィールドSA が、リファレンスから外れずに顧客業務デモを短時間で作れる）を追う。
過去設計を骨格として復活させる案（A）と、まっさらに再設計する案（C）を検討し、**C を採る**
（施主の判断、2026-07-06）。

## 決定

1. **フレッシュ再設計（C）**。ADR-0013 の 3層プラグイン／中央レジストリ前提のアーキテクチャに縛られない。
   ADR-0013 は本 ADR で **Superseded** とする。main 生存資産（認証・usecases owner/visibility・manifest 署名・
   file_search/ベクタストア・サンプルスキーマ実体化）は合う所だけ日和見的に流用する。
2. **能力モデル = 既存 JetUse 能力の組み合わせのみ**。生成するのは**フロント + データ**。バックエンドは
   JetUse 固定（デモ専用バックエンドを生成しない）。
3. **実行時のデモ = 静的SPAバンドル**（デモごと）。JetUse が `/api/demos/{id}/...` で配信し、ブラウザから
   ユーザー認証 + デモスコープで JetUse API を叩く。デモ専用サーバコードは持たない。秘密・外部接続は
   共有 `connector.invoke` 能力で解く（デモ専用サーバではない）。
4. **サブプロジェクト分解**: SP1 JetUse API（共通・土台）／SP2 テナンシ+Demo エンティティ（Internal）／
   SP3 ビルダー（Internal）／SP4 マーケットプレイス（Internal・将来）。詳細は specs/17。
5. **テナント = ユーザー**（Identity Domains）。デモは owner 単位で分離・デフォルト非公開・将来のマーケットで
   公開選択したもののみ他者可視。
6. **IaC は生成しない**（Galley 的 IaC 生成/検証は不採用。ADR-0013 §6 の判断を踏襲）。プラットフォームは
   既存 JetUse 基盤の上でのみ動く。

## 影響

- specs/16（PLG-01 manifest）と ADR-0013 の 3層/レジストリ/Platform API ブローカーの枠組みは**規範ではなくなる**。
  manifest 署名の実装は SP4 の署名土台として再利用しうる。
- SP1 は `main` 発（Public 共通）。SP2〜4 は `dev` 発（Internal 固有）。ADR-0014/0016 のブランチ運用に従う。

## 非ゴール / 留保

- SP2〜4 の詳細設計は各 SP の仕様で確定する。
- 本 ADR の採択自体が人間ゲート。承認まで実装着手しない。

## 代替案

- **A: 過去設計（ADR-0013）を骨格として復活**。最速だが、リセットで失われた前提の再構築コストと、
  宣言型 L1 中心の設計が「グラフィカルなフロント生成」という現行構想と噛み合わない。却下。
- **B: 骨格流用しつつビルダーを最初からフロントコード生成に振る**。C とほぼ同帰結だが、ADR-0013 の
  用語/制約に部分的に縛られ整理が濁る。まっさら（C）の方が明快。
