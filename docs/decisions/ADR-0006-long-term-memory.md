# ADR-0006: 長期メモリはOCIネイティブ機能（プロジェクトLTM + memory_subject_id）で提供する

日付: 2026-06-11 / 状態: **Accepted**（初版「アプリ実装案」はユーザー指摘による再調査で差し替え） / 関連: SPIKE-10, SPIKE-10b, specs/11 [AGT-05]

## 経緯

1. 初版: SPIKE-10でsubject_idパラメータが機能せず「長期メモリAPIは存在しない」と誤判定 → アプリ実装案を起案
2. **ユーザー指摘**「長期/短期メモリはプロジェクト作成時に指定しないと後から使えない仕様のはず」→ 再調査（SPIKE-10b）
3. 実機+公式docsで確定: ①プロジェクトの `long-term-memory-config` は**作成時のみ設定可**（更新は明示エラー） ②利用は **`memory_subject_id` を会話metadataに付与**（responsesパラメータではない） ③メモリ有効プロジェクトで**会話横断の想起・subject分離が実機動作**

## 決定

- **OCIネイティブ長期メモリを採用**。`jetuse-dev-project`（LTM: extraction=gpt-oss/embedding=cohere.embed-v4.0、STM condenser、retention 720h/168h）へアプリを移行
- アプリは会話作成時に `memory_subject_id` = JWT sub を付与（コード変更はこの1点）
- 初版のアプリ実装案（USER_MEMORIESテーブル）は**棄却**。理由: プラットフォーム機能で要件（会話横断パーソナライズ・subject分離・retention）が満たせ、抽出・注入の自前運用コストが不要

## 残課題（Phase 8ガバナンスへ）

- ~~記憶の個別削除API未発見~~ → **会話削除で派生記憶も消えることを実機確認**（AGT-05.md追補）。CHAT-09の削除同期が忘れられる権利の主経路として機能。ただし事実が複数会話で言及されている場合は全言及会話の削除が必要 — 完全消去の保険としてsubjectローテーション案は保持
- 抽出された記憶の内容をユーザーに開示するUI（記憶の透明性）

## 教訓

- OpenAI互換APIの未知フィールドは黙殺される（SPIKE-10）— 機能探索は対照実験必須
- **プロジェクト作成時オプションは作成前に全て棚卸しすること**（LTM/STM/retentionすべて作成時のみ。再作成はRAGファイル再取り込み等の移行コストを伴う）
