# ADR-0014: Platform API ブローカーの認可モデル（スコープ／短期トークン／テナント境界／監査）

日付: 2026-06-27
状態: **採用**（2026-06-27 施主承認。レビューで3点を反映。PAPI-01 で起票）

> 本 ADR は施主レビュー（2026-06-27・stage-runner Wave 1 報告）で **採用**された。承認にあたり
> ①監査の「保証」を best-effort 記録へ文言修正、②発行粒度の確定を PAPI-02 へ明記、
> ③L3 コンテナは秘密鍵を持たずトークンのみ提示、の3点を反映した（下記該当節）。
> `docs/enhance/202607-demo-platform-plan.md` §11 が予告した「ADR-0014（S3）: Platform API 認可モデル」。

## 背景

デモ生成プラットフォーム（ADR-0013）では、L2 コネクタ・L3 ホスト型アプリ・生成デモが
**テナントデータ**（ADB の業務データ・会話・文書・コネクタ）へ到達する必要がある。素朴に DB 接続情報を
配ると、(1) テナント越境、(2) 最小権限の崩壊、(3) 監査不能、(4) レート制御不能、という越境リスクが生じる
（plan §12「越境・最小権限（最重要）」）。

plan §7 / D5 は解として **Platform API ブローカー**を置く：「DB 認証情報を持たずにテナントデータへ到達する
**唯一の正規経路**」。プラグインはブローカーが発行する**スコープ付き短期トークン**を提示し、ブローカーが
スコープ・テナント境界・監査・レート制限を一元的に強制する。本 ADR はその**認可モデル**を固定する。
実 API ルート（rag.search/db.query 等の本体）は PAPI-03、発行・承認フローは PAPI-02。

## 決定

### 1. スコープ語彙（最小権限の単位）

スコープは plan §7 の例に揃え、manifest 検証（`jetuse_core.plugins.manifest.PLATFORM_SCOPES`）と
**同一集合**を正本とする（manifest の `permissions` と発行トークンの `scope` が必ず突き合う）：

| スコープ | 許可される操作 |
| --- | --- |
| `platform:rag.search` | RAG（File Search）検索（読取） |
| `platform:db.query` | 業務 DB の**読取**照会（NL2SQL の SELECT 実行先） |
| `platform:conversations.read` | 会話履歴の読取 |
| `platform:files.read` | 文書・ファイルの読取 |
| `platform:files.write` | 文書・ファイルの書込 |
| `platform:connector.invoke` | L2 MCP コネクタの呼び出し |

- スコープは**追加のみで後方互換**に拡張する（新語彙は manifest 側と同時に増やす）。
- 付与スコープは常に `PLATFORM_SCOPES` の**部分集合**でなければならない（未知スコープは発行・検証で拒否）。

### 2. 短期トークン（テナント=Project OCID・プラグインID・付与スコープ内包）

呼び出しごとに JetUse（＝ブローカー）が**短期 JWT** を発行する。JetUse が発行も検証も担う閉じた境界
（自分が自分を検証する）なので、スパイク／MVP は**対称鍵 HS256**を採用する。鍵は `platform_broker_secret`
として .env / Vault から注入し、**コミットしない**。

トークンの主張（claims）：

| claim | 意味 |
| --- | --- |
| `iss` | `jetuse-platform-broker`（発行者固定） |
| `aud` | `jetuse-platform-api`（受け手＝ブローカー API 固定） |
| `sub` | プラグインID（呼び出し主体の同定。manifest `id`） |
| `tenant` | **テナント境界 = Project OCID**（このトークンで触れてよいテナント） |
| `scope` | 付与スコープ（OAuth2 慣習でスペース区切り。`PLATFORM_SCOPES` の部分集合） |
| `jti` | トークン一意ID（監査・失効の継ぎ目） |
| `iat` / `nbf` / `exp` | 発行・有効化・失効時刻。TTL 既定 **300 秒**、上限 **900 秒** |

- スコープは manifest `permissions` 由来で、**インストール／合成時に承認**された範囲だけを載せる（PAPI-02）。
- TTL を短く保ち失効を時間で担保する（MVP は失効リスト無し。`jti` は将来の失効・重複検知の継ぎ目として残す）。
- **発行粒度は PAPI-02 で確定する**（呼び出しごと／セッションごと）。MVP は TTL 内の単回使用強制（リプレイ防護）を
  持たないため、粒度がそのまま**リプレイ露出窓**に影響する。粒度を細かく（呼び出しごとに発行）するほど露出は小さい。
  リプレイリスト／単回 `jti` 消費の要否は PAPI-02 で粒度確定とあわせて再判断する。

### 3. テナント境界の強制（越境防止）

ブローカーは API 呼び出しのたびに **トークンの `tenant` と要求リソースのテナントが一致**することを必須にする。
不一致は `tenant_mismatch` で拒否し**監査に DENY を残す**。これが plan §12 の「データはインスタンス所有・
アクセスは仲介経由」の実装中核であり、L3（コンテナ）にもベースURL＋短期トークン注入で同じ強制を効かせる
（D3 の解）。

**L3 コンテナは検証鍵（`platform_broker_secret`）を保持しない**。コンテナへ注入するのは*短期トークン*のみで、
検証は常に JetUse（ブローカー）側で行う。よって対称鍵 HS256 でも秘密鍵が外部へ漏れず安全側を保てる
（検証主体が JetUse 外へ分散したら非対称鍵へ移行＝代替案）。

### 4. fail-closed 検証

トークン検証は**あらゆる失敗を「不可」に倒す**：署名不正・期限切れ・`nbf` 未到来・`iss`/`aud` 不一致・
未知スコープ混入・`tenant` 欠落・鍵未設定（`platform_broker_secret` 空）・その他内部エラーのいずれも
**例外を漏らさず拒否**する（`manifest.verify_signature` の fail-closed 方針と整合）。鍵が未設定の環境では
発行も検証もできない＝**安全側に閉じる**。

### 5. 監査とレート制限

- **監査**: 全アクセス（許可・拒否とも）を `platform_broker_audit`（テナント・プラグイン・スコープ・
  許否・理由・jti・時刻）に**ベストエフォートで記録する**（記録失敗でサービスを止めない＝`audit.py` と同方針）。
  既存 `audit.py`（機能別集計）とは別表とし、越境の試行（DENY）を監査に残すことを意図する。
  ただし MVP では監査書込はベストエフォートのため、書込失敗時に DENY 記録が欠ける可能性がある。
  **DENY イベントを監査できないときに拒否を fail-closed にするか（＝監査必須化）は PAPI-03 で再検討する**
  （可用性とのトレードオフ。本 ADR では「監査を一元的に残せる位置にブローカーがある」ことを要件として固定）。
- **レート制限**: ブローカーは将来 (tenant, plugin, scope) 単位でレートを絞る継ぎ目を持つ。**スパイクでは
  設計のみ**（PAPI-03 で実装）。本 ADR では「ブローカーが一元的に絞れる位置にある」ことを要件として固定する。

### 6. スコープ（このADR/PAPI-01 の範囲）

- **やる**: 認可モデルの確定＋スパイク（発行・検証・スコープ強制・テナント一致・監査表）。
- **やらない（後続）**: スコープ承認 UI／発行フロー（PAPI-02）、実 API ルート本体（PAPI-03）、
  OIDC 連携での発行主体認証（INFRA-02）、失効リスト／非対称鍵（将来 — §代替案）。

## 代替案

- **DB 認証情報を直接配布**: 越境・最小権限・監査・レート制御がすべて崩れる（plan §12 が明確に禁止）。不採用。
- **長期 API キー**: 失効が運用依存になり越境時の被害時間窓が長い。短期トークン＋時間失効を採用。
- **非対称鍵（RS256/ed25519）でのトークン**: 検証を外部（L3 コンテナ等）へ委譲して鍵共有を避けられる利点。
  ただし MVP は JetUse 内で発行＝検証が閉じるため過剰。**将来、検証主体が分散したら移行**（`alg` を差し替え
  られるよう発行/検証は alg を一点に集約してある）。
- **OPA/外部ポリシエンジン**: スコープ語彙が小さい現段階では過剰。スコープ＋テナント一致の単純規則で足りる。
