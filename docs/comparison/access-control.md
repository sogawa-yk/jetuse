# 比較: アクセス制御（IP制限・レート制限）の方式（SEC-03）

API GWの背後に置いた本構成で「IP制限・レート制限」をどう実装するかの比較。
採用 = **API Gatewayのレート制限ポリシー**（実装・実機検証済み）。IP制限はWAF/NSGを比較に留める。

## レート制限

| 方式 | 評価 | 状態 |
|---|---|---|
| **API GW レート制限ポリシー（採用）** | デプロイメント仕様に `rate_limiting`（rate_in_requests_per_second + rate_key）。追加コストなし、設定のみ | **実装・本番稼働**（送信元IP単位20 req/s） |
| アプリ内レート制限（slowapi等） | CIに状態を持たせる必要・Fnでは不可・水平スケールで破綻 | 不採用 |
| WAF レート制限 | より高機能（地理・ボット）だが有料・本プロトタイプには過剰 | 比較のみ |

- `rate_key`: **CLIENT_IP**（送信元IP単位）を採用。`TOTAL`（全体）も選択可（`rate_limit_key`変数）
- 注意: 社内が単一NAT/プロキシ経由だと全員が1 IP扱いになり得る → その場合は値を上げるか
  `TOTAL`へ切替、または認証済みsubキーが必要なら usage plan（APIキー方式）を検討（将来）
- 実機: 60並列バーストで **36×200/401 + 24×429** を観測（閾値超過分が429になることを確認）

## IP制限（送信元の絞り込み）

| 方式 | 評価 |
|---|---|
| **NSG/セキュリティリスト**（採用候補） | GWのNSGで ingress を社内CIDR/VPNに限定。L4・無料・即時。本番化時に社内レンジを設定するのが素直 |
| WAF アクセスコントロール（IP allowlist/国制限） | L7で柔軟だが有料。グローバル公開+細かい制御が要る場合に |
| API GW単体 | ネイティブのIP allowlistルールは持たない（CORS/認証/レート/ mTLSのみ） |

- 本プロトタイプは**認証（OIDC/JWT）で実質的なアクセス制御済み**のため、IP制限は本番要件化したら
  NSGのingress CIDR制限で対応する方針（手順はPhase 12のデプロイガイドに記載予定）

## 設定（Terraform）

- `rate_limit_rps`（既定0=無効、dev=20）/ `rate_limit_key`（既定CLIENT_IP）= `infra/terraform/.../variables.tf`
- 0にすればポリシー自体が付かない（`dynamic block`）
