# 計画: AWS版との差分「簡易版ギャップ(B項目)」の解消

対象は `docs/comparison/aws-reference.md`「未充足ギャップ B. 簡易版」の5項目。
**各項目は必ず「実現可能性調査(SPIKE)」→ ゲート判定 → 実装」の順で進める**。
調査の結果「OCIプラットフォーム制約で不可」と確定したものは、無理に実装せず
A項目（プラットフォーム制約）へ降格して記録する。

タスクIDは `GAP-NN`。1タスク=1ブランチ→main。実機検証主義（レポートは `docs/verification/`）。

## 全体の進め方とゲート

```
Phase B-investigate（調査優先・並行可）
  ├ GAP-01 出力ガードレール        … 調査軽（実装容易の確証取り）
  ├ GAP-03 Code Interpreter相当    … 調査重（サンドボックス安全性が肝）
  └ GAP-05 全二重音声              … 調査重（OCIにモデルがあるか＝go/no-go）
      ↓ 各ゲートでgo/no-go判定（ユーザー承認）
Phase B-implement（goになったものだけ）
  ├ GAP-01 → 実装
  ├ GAP-02 SAML実機検証（人間IdP作業が主、エージェントは事前準備・検証支援）
  ├ GAP-03 → 実装（goなら）
  ├ GAP-04 AgentCore相当UC統合
  └ GAP-05 → 実装（goなら）
```

優先順（コストと確実性）: **GAP-01 → GAP-02 → GAP-04 → GAP-03 → GAP-05**。
（前2つは低コスト・高確度、後2つは調査結果次第）

## GAP-01: 出力ガードレール ✅ 完了(マネージド導入分=プロンプトインジェクションのみ。CM/PIIはマネージド非対応で未実装)

**目的**: 入力モデレーション(SEC-02)に加え、LLMの**出力**も不適切判定し、必要なら遮断/マスクする。

### 実現可能性調査（SPIKE-G1）
- ストリーミング出力(SSE)へのモデレーション適用方式を比較:
  1. **事後判定**（全文生成後にチェック→NGなら差し替え。実装容易だがTTFTの利点を消す）
  2. **センテンス単位**（確定文ごとに高速モデルで判定しながら流す。VOICE-03のセンテンス分割を流用）
  3. **バッファ＋遅延フラッシュ**（N文字バッファしてから判定して放出）
- 高速モデル(llama-3.3-70b, TTFT 0.07s)で1文あたりの判定レイテンシを実測
- 既存の `jetuse_core/moderation.py`（入力用）を出力にも使えるか、プロンプト調整の要否
- 成果物: `docs/verification/SPIKE-G1.md`（方式比較＋レイテンシ実測）
- **ゲート**: 体感を損なわない方式が成立するか（センテンス単位が有望と想定）

### 実装
- `moderation.check_output` 追加、`chat/stream` のdelta経路に組み込み（`OUTPUT_MODERATION_ENABLED`、既定off）
- 遮断時は以降のdeltaを止め監査(`moderation_block`/方向=output)に記録
- 完了条件: 不適切出力が遮断され、正常出力は素通り。レイテンシ増を実測レポート

### リスク/見込み
- **高確度・低〜中コスト**。SEC-02の機構をほぼ流用。主リスクはUX（遅延）→ センテンス単位で吸収

## GAP-02: SAML認証の実機検証 ✅ 調査完了(マネージド完結=go実証。CLI構成可、最終ブラウザSSOは対話操作。検証用ドメインは削除済み)

**目的**: SEC-01で起草済みの手順書(`docs/setup/saml-federation.md`)を実機で検証し「稼働」に格上げ。

### 実現可能性調査（SPIKE-G2）
- テスト用IdPの選択肢調査: ①社内Entra/Googleのテストテナント ②無償SAML IdP（無理なら）
  ③Identity Domain同士のSAMLフェデレーション（OCI内で完結する検証）
- **OCI内完結の検証経路**が取れるか（別Identity DomainをIdPに見立てる）を最優先で確認
  → 取れれば人間のIdPテナント準備を待たずにエージェント主導で検証可能
- 成果物: `docs/verification/SPIKE-G2.md`（検証経路の決定）

### 実装/検証
- 調査でOCI内完結が可能なら: テスト用Domain作成→SAML設定→E2E（手順書のチェックリスト消化）
- 不可なら: 人間のIdPテナント作業待ち（エージェントは設定値の事前生成・検証スクリプト準備）
- 完了条件: 手順書のチェックリスト（未割当拒否・JIT作成・切り戻し・sub安定）が実機で全項目green

### リスク/見込み
- **高確度だが外部依存**。OCI内完結経路が取れるかが鍵。取れなければ人間待ち項目のまま

### リスク/見込み

## GAP-03: Code Interpreter相当（SDK/LangGraphエンジン対応） ⛔ no-go確定(SPIKE-G3。マネージドのコード実行はResponses built-in=nativeのみ。SDK/LangGraph経路は不可→「コード実行が必要ならnative」を使い分け確定。B→A降格)

**目的**: nativeエンジン限定（Responses built-in）のコード実行を、Agents SDK/LangGraphでも使えるようにする。

### 実現可能性調査（SPIKE-G3）★安全性が最重要
- 方式比較:
  1. **OCI Code Interpreter built-inの再利用可否**: SDK/LangGraphはchat completions経由のため
     Responses built-inは使えない（ADR-0007）。回避策があるか確認
  2. **自前サンドボックス実行ツール**: function toolとしてPythonを隔離実行
     （候補: 別コンテナ/gVisor/サブプロセス+リソース制限/制限付きインタプリタ）。
     **任意コード実行のセキュリティ評価が必須**（ネットワーク遮断・CPU/mem/時間制限・FS隔離）
  3. **OCI Functionsを使い捨て実行環境にする**案の可否
- 成果物: `docs/verification/SPIKE-G3.md`（方式比較＋セキュリティ評価＋ADR案）
- **ゲート**: 安全に実行できる方式が現実的か。**安全性に確信が持てなければ実装しない**
  （その場合「コード実行が必要ならnativeエンジンを選ぶ」を正式な使い分けとして確定）

### 実装（goの場合のみ）
- サンドボックス実行ツールを `tools.py` に追加（requires_approval=True 必須）
- SDK/LangGraphエンジンから利用可能に
- 完了条件: 隔離環境でコード実行→結果反映。脱獄/リソース枯渇の防御を検証レポートで実証

### リスク/見込み
- **調査結果次第。セキュリティリスクが高くno-go妥当性も十分ある**。チェックポイント④の観点とも整合確認

## GAP-05: 音声チャットの全二重化 ⛔ 真の全二重はno-go確定(SPIKE-G5。OCIにspeech-to-speechモデル無し→B→A降格。連続半二重=VAD自動送信+簡易バージインはフロント改善としてbacklog)

**目的**: 半二重(VOICE-03: 押して話す)を、JetUseのNova Sonic相当の全二重(割り込み可・連続対話)へ近づける。

### 実現可能性調査（SPIKE-G5）★プラットフォームのgo/no-goが肝

## 追加ユースケース: マルチエージェント構成の活用（MA-01）※計画のみ・未実装

GAP-04でマネージドなエージェント実行基盤(Hosted Application / LangGraph)が使えるようになったのを機に、
**「マルチエージェント構成でしか質が出ない」ユースケース**を1本作る。現状のサンプル
（researcher→summarizerの2ノード）は単一ループでも代替可能で価値が薄いため、これを置き換える。

### ユースケース案: 社内横断リサーチ＆提案メモ生成

入力例: 「競合A社の新製品動向と、自社の該当カテゴリ売上を踏まえた提案メモを作って」

1指示で、**異種ソースを並列に当たり → 役割分担で検証・統合**する。

### なぜマルチエージェントでないとダメか（単一ループで劣る理由）

- **異種ソースの並列探索**: 社内文書(RAG) / 売上DB(NL2SQL) / Web検索 を**同時**に走らせる。
  単一ReActループは直列でしか回せず遅く、文脈も混線しやすい
- **役割分担による品質**: リサーチ専門 / 検証専門(クリティック) / 執筆専門 を分けると、
  「自己批評」より高品質になる（別エージェントによる敵対的裏取り）
- **グラフ構造が本質**: 分岐・並列・合流・条件付きループ(不足なら再探索)はLangGraphの土俵

### 構成（LangGraphグラフ）

```
[Planner] 質問を3サブ課題へ分解
    ├─並列─> [社内文書リサーチャー]  (既存 rag_search)
    ├─並列─> [DBアナリスト]          (既存 query_database / NL2SQL)
    └─並列─> [Web リサーチャー]       (既存 web_search / web_fetch)
              ↓ 合流
        [クリティック] 各主張の根拠・矛盾を敵対的に検証(裏が取れない主張は要再探索→Plannerへ条件ループ)
              ↓
        [ライター] 引用付きで提案メモに統合
```
- 既存ツール資産(RAG / NL2SQL / Web)を**専門エージェントに割り当てて再利用**できるのが強み
- 実行基盤: GAP-04のHosted Application(LangGraphグラフを差し替え)で常設＝マネージド軸。
  またはインプロセスLangGraphエンジン。アプリからは1ユースケースとして実行

### 実現可能性調査（着手時のSPIKE-MA1）

- 並列ノードからの異種ツール呼び出し（rag_search/NL2SQL/Web）が、OCI互換LLM+LangGraphで安定動作するか
- クリティックの「裏取り失敗→再探索」条件ループが暴走しない上限設計(再帰上限/ラウンド上限)
- レイテンシ実測（並列化の効果）と、Hosted常設 vs インプロセスのコスト比較
- ゲート: 体感許容のレイテンシで、単一ループより明確に良い出力が出るか

### 実装範囲（goの場合）
- LangGraphの本格グラフ(上記)をエージェント化し、Hosted Applicationのコンテナを差し替え or
  インプロセスエンジンに専用ユースケースを追加
- アプリUIに「横断リサーチ」ユースケースを追加、引用付きレポートを表示
- 完了条件: 異種3ソース並列→検証→統合のE2E、検証レポート docs/verification/MA-01.md

### 位置づけ
- B項目(ギャップ解消)ではなく、**OCIマネージド・エージェント基盤の差別化デモ**(プリセールス価値)
- 現サンプル(researcher→summarizer)の正式な置き換え先
- **本項は予定の記載のみ。実装は別途ゲート承認後**
